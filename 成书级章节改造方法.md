# 成书级章节改造方法

> 本文档用于指导《模仿学习的数学原理》后续章节的成书级改造。
>
> 第一篇已经完成阶段性改造，本文不再记录第一篇的具体过程，也不再维护某一章的专属模板，而是沉淀为后续全书通用的改造规则。
>
> **公式格式统一由《公式格式说明.md》管理。本文只规定章节结构、写作方法、数学主线、公式推导主线、命题证明、受众定位、读者可读性和成书级检查逻辑。**

---

## 1. 本书的四条顶层约束

后续所有章节改造，都必须同时满足四条约束。

### 1.1 数学对象主线必须推进

本书不是模仿学习方法清单，而是围绕一条数学对象主线展开：

```text
专家轨迹
→ 观测与动作
→ 策略
→ 专家策略与学习策略
→ 训练目标
→ 专家分布与执行分布
→ 分布偏移与误差累积
→ 序列决策、轨迹分布、长期目标
→ 数据聚合、分布匹配、概率建模、生成式策略
```

每章都要回答：

```text
本章推进了哪一个数学对象？
本章引入了哪一个新的训练目标、分布视角或优化问题？
本章和前后章节的数学关系是什么？
```

如果某一节只是介绍方法现象，但没有推进数学对象、训练目标或数学问题，就需要重写。

### 1.2 公式推导主线必须显性化

《模仿学习的数学原理》不能只解释公式“是什么意思”，还要解释公式“为什么会出现”和“为什么成立”。

每章都应尽量形成一条公式推导主线：

```text
上一章公式留下的问题
→ 本章核心公式从哪里来
→ 本章公式解决了什么问题
→ 本章公式暴露了什么局限
→ 下一章为什么需要新的公式
```

例如第二篇可以形成如下公式主线：

```text
BC 专家分布目标
→ 策略闭环执行会诱导轨迹分布 p_pi(tau)
→ 单步 loss 与轨迹级目标 J(pi) 不一致
→ 状态转移 P(s_{t+1}|s_t,a_t) 描述动作如何改变未来
→ MDP 提供序列决策框架
→ 轨迹分布解释为什么长期目标不能只看单步动作误差
```

章节中建议保留一个模块：

```markdown
## 本章公式主线
```

这个模块不必很长，但必须回答：

```text
本章核心公式从前一章哪个问题推出来？
它解决了什么问题？
它又留下了什么问题？
```

### 1.3 关键结论必须有推导或证明

本书讲的是“数学原理”，不是只讲“公式解释”。关键结论不能只给出结论，必须尽量给出命题、推导或证明。

不是所有公式都要证明，但以下类型必须优先证明：

```text
会改变工程判断的结论；
连接两个章节主线的结论；
解释某个方法为什么失败的结论；
说明某个训练目标为什么等价于另一个目标的结论；
引出后续方法的关键数学结论。
```

优先补充证明的典型结论包括：

```text
最大似然等价于最小化负对数似然；
固定方差高斯负对数似然等价于 MSE；
MSE 的最优预测是条件均值；
BC 的误差累积风险；
DAgger 修正的是访问状态分布而不是简单换 loss；
MDP 中策略如何诱导轨迹分布；
单步模仿损失和轨迹级目标为什么不等价；
CVAE 的 ELBO 推导；
KL 项约束 posterior 与 prior 的关系；
Diffusion 去噪目标与 score / 条件动作分布的关系；
Flow Matching 速度场目标的基本推导。
```

证明必须服务于读者理解和工程判断，不是为了炫技。

### 1.4 受众是普通工程师

本书主要写给有工程背景、数学基础一般、希望理解模仿学习本质的算法工程师和学习者。

默认读者具备：

```text
基本机器学习概念；
基本监督学习、loss、训练集、验证集概念；
一点 PyTorch / 算法工程经验；
但不默认熟悉概率统计、强化学习理论、机器人学习理论。
```

因此章节写作必须做到：

```text
公式出现前先说明工程问题；
公式出现后逐项解释符号；
关键结论要有推导或证明；
证明不能跳步到“显然”“可得”“标准结论”；
抽象概念必须落回机械臂、机器人或策略学习场景；
不要用高级理论术语替代解释；
不默认读者熟悉概率统计、强化学习或机器人学习背景。
```

### 1.5 读完必须能形成工程判断

每章不仅要让读者“看懂”，还要让读者“能判断”。

例如读完某章后，读者应该能判断：

```text
BC 验证集 loss 很低但闭环失败，可能是训练分布和执行分布不一致；
MSE 在多模态动作上失败，可能是动作分布被平均；
DAgger 不是换损失函数，而是在修正学习策略访问分布的数据覆盖问题；
MDP 不是 RL 专属概念，而是描述策略如何通过动作改变未来输入的数学框架；
Diffusion Policy 不是简单换网络，而是在建模条件动作分布或动作序列分布。
```

后续章节建议保留一个模块：

```markdown
## 读完本章，你应该能判断什么？
```

这个模块用于把公式、证明、概念和工程决策显性连接起来，避免章节变成“知识正确但读者不知道有什么用”。

## 2. 成书级章节改造目标

成书级章节改造不是简单润色，也不是扩写字数，而是让每一章具备以下能力：

1. **概念递进清楚**：先定义基础对象，再引入依赖它们的复杂对象。
2. **数学符号清楚**：第一次出现的变量、公式、条件概率、损失函数都要解释。
3. **公式主线清楚**：每章要说明核心公式从哪里来、解决什么问题、留下什么问题。
4. **关键证明清楚**：关键结论要有命题、推导或证明，不能只写结论。
5. **工程含义清楚**：每个核心公式和关键证明都要能落回机械臂抓取与放置主线任务中理解。
6. **章节结构稳定**：章标题、小节标题、公式主线、公式索引、定义索引、思考题、推荐阅读形成统一格式。
7. **网页显示稳定**：公式、图片、表格、列表都要适合 mdBook 和 GitHub 网页阅读。
8. **读者路径友好**：面向普通工程师，不假设读者已经熟悉概率统计、强化学习或机器人学习背景。
9. **章节接力自然**：每章结尾要说明本章解决了什么，又为什么自然引出下一章。

一句话概括：

```text
先让读者知道问题从哪里来，
再定义概念，
再写公式，
再说明公式如何推出来，
再证明关键结论，
再落到工程任务，
最后指出局限和下一章问题。
```

## 3. 第一篇改造经验沉淀

第一篇改造形成了三类可复用经验。

### 3.1 专业性增强：从“讲概念”升级为“讲数学原理”

专业性增强不是堆更多术语，而是把概念背后的数学对象、目标函数、隐含假设和失败条件讲清楚。

第一篇中的典型做法包括：

```text
增加公式主线；
增加定义；
增加命题；
增加证明思路；
增加简单推导；
增加公式索引；
增加符号解释。
```

例如：

```text
连续动作常用 MSE
```

不能只停留在工程经验，而要进一步说明：

```text
固定方差高斯策略
→ 负对数似然 NLL
→ 与 MSE 等价
→ MSE 隐含单峰固定方差假设
```

再例如：

```text
BC 会分布偏移
```

不能只停留在现象描述，而要写成：

```text
BC 训练目标在 d^{pi_E} 上取期望；
部署执行目标在 d^{pi_theta} 上取期望；
问题不是 loss 一定写错了，而是期望所在的分布变了。
```

经验原则：

> **不要只告诉读者一个方法会怎样，而要把它背后的数学对象、目标函数、隐含假设和失败条件写清楚。**

### 3.2 可读性增强：从“作者知道”改成“读者能跟上”

可读性增强不是降低专业性，而是把读者的理解路径补完整。

每章要主动回答读者可能卡住的问题：

```text
为什么要引入这个概念？
这个公式到底在说什么？
这个符号在工程里对应什么？
我是不是可以先不用完全懂？
这个理论结论是不是在真实系统里严格成立？
这一章和下一章有什么关系？
```

例如，$d^{pi_E}$ 和 $d^{pi_theta}$ 不能只叫“专家分布”和“学习策略分布”，还要翻译成工程语言：

```text
d^{pi_E}：专家正常操作时，系统经常看到的画面、姿态、相对位置和状态集合。
d^{pi_theta}：模型自己控制以后，实际会把系统带到的画面、姿态、相对位置和状态集合。
```

再例如，$O(T^2 epsilon)$ 不能让读者误解成实机失败次数预测，而要说明：

```text
它不是精确失败次数预测；
它是风险量级提示；
它提醒早期错误会影响后续输入；
它提醒开环 loss 不能直接推出闭环稳定。
```

经验原则：

> **公式不是越多越好。真正重要的是让读者知道公式为什么出现、怎么读、对应什么工程现象、不能怎么误读。**

### 3.3 主线一致性增强：让章节之间形成“问题接力”

章节不能只是知识点并列摆放，而要形成问题递进链。

第一篇形成的接力关系是：

```text
第1章：模仿学习真正学的是策略，不是轨迹。
第2章：既然要学策略，最朴素的方法是把专家动作当标签做监督学习，这就是 BC。
第3章：但 BC 只在专家分布上训练，闭环执行时策略会进入自己诱导出的分布，于是出现分布偏移和误差累积。
第4章：既然问题来自学习策略自己会走到专家数据没有覆盖的状态，那就让学习策略跑出来，再请专家标注这些状态，这就是 DAgger。
```

经验原则：

> **每一章都应该回答上一章留下的问题，并自然引出下一章。章节之间要像接力棒，而不是像 PPT 目录。**

### 3.4 数学补充必须服务主线

i.i.d. 的补充不是为了单独讲概率统计，而是补上了第一篇主线中的关键断点：

```text
普通监督学习为什么通常可以依赖固定训练集？
BC 为什么训练形式像监督学习？
BC 为什么部署时又不能完全按普通监督学习理解？
```

补充后，第一篇主线更完整：

```text
普通监督学习
→ i.i.d. / 训练测试近似同分布
→ BC 借用监督学习形式
→ 但 BC 输出动作
→ 动作改变未来输入
→ 训练/执行分布不一致
→ 分布偏移
→ 误差累积
→ DAgger
```

经验原则：

> **数学补充不能为了补而补。只有当它能解释主线中的关键断点时，才值得加入正文。**

## 4. 可复用章节改造模板

后续改造第二篇、第三篇以及更后面的章节时，统一按下面七步检查和改造。

### 4.1 本章角色

先判断这一章在全书中的作用。

必须回答：

```text
这一章在全书主线里解决什么问题？
它承接上一章哪个问题？
它为下一章埋下什么伏笔？
如果删掉这一章，全书主线会断在哪里？
```

写作时要避免：

```text
只说“本章介绍 MDP”；
只说“本章介绍某个算法”；
只说“本章讲一个经典概念”。
```

更好的写法是：

```text
为了理解策略为什么会改变未来输入，我们需要一个描述闭环环境的数学框架，这就是 MDP。
```

### 4.2 数学对象

列出本章新增或强化的数学对象。

必须回答：

```text
本章新增哪些符号？
这些符号分别表示什么？
它们和上一章已有符号是什么关系？
它们在工程系统中对应什么？
后续章节会如何复用它们？
```

建议表格格式：

| 数学对象 | 符号 | 一句话含义 | 工程对应物 | 后续作用 |
|---|---|---|---|---|
| 状态 | $s_t$ | 环境真实状态 | 物体位姿、接触状态、机器人状态 | 定义 MDP 和轨迹分布 |
| 动作 | $a_t$ | 策略输出控制 | 末端增量、夹爪命令 | 改变下一时刻状态 |
| 转移 | $P(s_{t+1}|s_t,a_t)$ | 动作如何改变未来 | 机器人动力学和环境响应 | 解释闭环分布来源 |

如果一个章节没有新增数学对象，也必须说明它如何重组或深化已有对象。

### 4.3 核心公式

每章选出 3–5 个真正核心的公式，不要机械堆公式。

必须回答：

```text
本章最关键的 3–5 个公式是什么？
每个公式为什么出现？
公式中每个符号是什么意思？
这个公式解决了什么问题？
这个公式暴露了什么局限？
这个公式会在后续哪一章继续使用？
```

每个核心公式按下面格式展开：

```text
动机：为什么需要这个公式？
公式：稳定、规范地写出公式。
读法：用自然语言读一遍。
符号解释：逐项解释每个符号。
直觉：它在表达什么关系？
工程含义：在机械臂/机器人系统中对应什么？
常见误解：读者容易怎么误读？
```

### 4.4 推导与证明

判断本章哪些地方不能只讲结论，必须给出推导、证明或命题化解释。

必须回答：

```text
有没有从 A 推到 B 的关键步骤？
有没有容易被当成结论但其实需要解释的地方？
有没有适合写成命题的小结论？
有没有理论量级结论需要说明适用边界？
```

推荐格式：

```markdown
> **命题 X.Y：结论名称**
>
> 在什么假设下，什么结论成立。

**证明思路**：先解释为什么要证明它，以及证明将如何展开。

**证明**：逐步推导，不跳过对普通工程师不显然的步骤。

**这个命题告诉我们什么？**：把数学结论翻译成工程判断。

**常见误解**：指出读者容易误读的地方。
```

如果证明过长，正文给核心推导，完整推导放到附录。

### 4.5 工程解释

每个抽象数学对象都要落回工程系统。

必须回答：

```text
这个数学对象在机械臂、机器人日志或自动驾驶系统里对应什么？
读者如何通过数据、图像、回放理解它？
如果这个假设不成立，工程上会出现什么失败？
这个公式能帮助工程师做什么判断？
```

尤其是下面这些对象必须给工程解释：

```text
状态；
观测；
动作；
轨迹；
策略；
状态转移；
访问分布；
轨迹分布；
长期目标；
occupancy measure；
动作分布；
隐变量；
去噪过程；
flow matching 速度场。
```

### 4.6 常见误解

每章至少主动澄清 2–4 个读者最容易误解的点。

必须回答：

```text
读者最可能误读什么？
这个公式有没有适用边界？
这个结论是不是理论量级，不是实机精确预测？
这个方法解决了什么，又没有解决什么？
```

建议格式：

| 常见误解 | 正确认识 |
|---|---|
| BC 有分布偏移，所以 BC 没用 | BC 是起点，不是终点，仍然是大规模行为数据学习的基础 |
| $O(T^2 epsilon)$ 是真实失败次数预测 | 它是最坏情形风险量级提示 |
| DAgger 就是多采数据 | DAgger 的关键是新数据来自学习策略访问分布 |
| MDP 是强化学习专属概念 | MDP 是描述闭环序列决策的通用框架 |

### 4.7 章节接力

每章结尾都必须自然引出下一章。

必须回答：

```text
本章解决了上一章留下的什么问题？
本章暴露了什么新的局限？
下一章为什么自然出现？
读者为什么应该继续读？
```

推荐结尾格式：

```text
本章说明了……
但这仍然没有解决……
因此下一章需要引入……
```

## 5. 第二篇改造特别提示

如果第二篇主题是 MDP、轨迹分布、序列决策和长期目标，不能写成普通强化学习教材，而要从第一篇自然延伸。

第一篇已经说明：

```text
BC 的问题是专家分布和学习策略分布不一致。
```

第二篇要继续回答：

```text
为什么策略执行会诱导出分布？
这个分布如何由状态转移、动作和环境决定？
单步动作损失为什么不能代表整个任务的长期成功？
为什么需要轨迹级目标？
```

因此，第二篇的主线不应该是：

```text
什么是 MDP；
什么是状态；
什么是奖励；
什么是价值函数。
```

而应该是：

```text
为了理解模仿学习的闭环问题，
我们需要一个描述“策略如何改变未来”的数学框架，
这个框架就是 MDP。
```

建议第二篇主线：

```text
第一篇：BC 暴露训练分布和执行分布错配。
→ 第二篇：引入 MDP 描述策略、动作和环境如何共同产生未来。
→ 策略执行诱导轨迹分布 p_pi(tau)。
→ 真正任务目标往往是轨迹级长期目标 J(pi)。
→ 单步模仿损失和长期任务成功之间存在差距。
→ 后续章节再讨论如何用更强的策略表示和数据闭环缩小这个差距。
```

第二篇常见读者卡点包括：

```text
状态和观测有什么区别？
MDP 为什么要引入？
轨迹分布是不是就是数据集？
长期目标和单步 loss 有什么区别？
reward 和 imitation loss 是什么关系？
为什么 trajectory-level objective 比 step-level loss 更难？
```

这些卡点要在正文中提前处理。

## 6. 章节改造的基本方法

### 6.1 从真实困惑进入

章节开头不能只说“本章介绍某方法”，而要回答读者为什么需要这一章。

推荐开篇回答：

```text
上一章留下了什么问题？
工程实践中会遇到什么困惑？
如果不懂本章，会在哪些判断上出错？
```

例如：

```text
如果我们已经知道 BC 在闭环中会分布偏移，那么下一步不是马上换一个更复杂的网络，而是要先问：策略为什么会改变未来输入？我们需要一个能描述这种闭环因果关系的数学框架。
```

### 6.2 按概念依赖链组织小节

章节小节顺序必须遵守概念依赖关系，避免“公式里已经出现的对象尚未定义”。

推荐链路：

```text
上一章问题
→ 本章真实困惑
→ 基础对象定义
→ 新数学对象
→ 本章公式主线
→ 训练目标或优化目标
→ 关键命题 / 推导 / 证明
→ 公式拆解
→ 工程解释
→ 失败模式或局限
→ 下一章问题
```

### 6.3 第一次出现的重要概念必须正式定义

凡是第一次出现、后续会反复使用的概念，都应该用“定义 X.Y”显式给出。

定义格式：

```markdown
> **定义 X.Y：概念名**
>
> 用一段简洁准确的话定义该概念。
```

定义之后必须接解释：

```text
这个定义里有哪几个关键词？
这些关键词分别对应什么工程对象？
为什么后续章节会反复用到它？
```

定义不宜过长。定义负责“准确”，后面的段落负责“解释”。

### 6.4 核心公式必须讲清楚

核心公式不能直接抛给读者，必须按下面顺序拆解：

```text
动机
→ 公式
→ 读法
→ 符号解释
→ 直觉
→ 工程含义
→ 常见误解
```

所有重要公式都要尽量遵守这个结构，尤其是：

```text
策略公式；
损失函数；
概率分布；
最大似然；
负对数似然；
期望；
KL 散度；
MDP 状态转移；
轨迹分布；
occupancy measure；
diffusion / flow matching 相关公式。
```

公式格式、GitHub 网页兼容性、mdBook 兼容性全部以：

```text
公式格式说明.md
```

为唯一准则。

### 6.5 每章必须有公式主线

公式主线不是简单列公式，而是说明公式之间的推进关系。

推荐格式：

```markdown
## 本章公式主线

上一章已经得到：……
但它留下的问题是：……
因此本章引入：……
这个公式说明：……
它解决了：……
但它仍然不能解决：……
所以下一章需要：……
```

### 6.6 关键结论必须写成命题、推论或证明

关键结论建议使用“命题 / 推论 / 证明”结构。

推荐格式：

```markdown
> **命题 X.Y：结论名称**
>
> 在什么假设下，什么结论成立。

**证明思路**：……

**证明**：……

**这个命题告诉我们什么？**：……

**常见误解**：……
```

注意：证明中的公式格式仍然必须遵守《公式格式说明.md》。不要因为证明较长，就使用 GitHub 不稳定的公式写法。

### 6.7 证明要分层，不要把正文写成纯数学论文

证明建议分三层：

```text
第一层：直觉推导；
第二层：正文中的核心证明；
第三层：附录中的完整证明或补充。
```

正文中优先证明那些能直接改变工程判断的结论。

过于基础或过长的概率推导可以放到附录，但正文必须告诉读者它为什么重要。

### 6.8 每章都要落回工程主线

本书采用“视觉引导机械臂抓取与放置任务”作为统一工程主线。

每一章都要尽量回答：

```text
本章的数学概念，在机械臂抓取与放置任务中解决了什么问题？
如果这个数学假设不成立，工程上会出现什么失败？
这个公式或证明能帮助工程师做什么判断？
```

不是每章都必须完全一致，但核心模块建议保留。


## 8. 标题与小节标题规则

### 8.1 章标题要表达知识推进路径

标题最好包含两个部分：

```text
主题对象 + 本章推进方向
```

例如：

```text
Behavior Cloning：从专家样本到监督学习策略
分布偏移：为什么训练误差小不代表执行成功
DAgger：用交互数据修正策略分布
MDP：描述策略如何改变未来的数学框架
CVAE：从隐变量动作模式到可训练生成策略
```

### 8.2 小节标题要专业但可读

推荐标题风格：

```text
概念名：本节回答的问题
```

例如：

```text
观测与状态：策略输入从哪里来
动作与动作空间：策略输出到哪里去
轨迹：专家行为的时间结构
公式主线：为什么从单步 loss 推到轨迹目标
命题：MSE 为什么会学到条件均值
```

不建议大量使用：

```text
第一层问题：……
第二层问题：……
第三层问题：……
```

这种写法适合草稿推理，但成书目录显得不够正式。

## 9. 公式主线、证明、公式索引与定义索引

### 9.1 公式主线

每章建议包含“本章公式主线”模块。

推荐格式：

```markdown
## 本章公式主线

上一章的核心公式是：……
它说明了：……
但它留下了：……
本章因此引入：……
本章核心公式说明：……
它为下一章铺垫：……
```

公式主线不是公式索引。公式索引回答“本章有哪些公式”，公式主线回答“这些公式为什么按这个顺序出现”。

### 9.2 命题与证明

关键结论建议写成命题、推论或证明。

推荐格式：

```markdown
> **命题 X.Y：结论名称**
>
> 在什么假设下，什么结论成立。

**证明思路**：……

**证明**：……

**这个命题告诉我们什么？**：……

**常见误解**：……
```

### 9.3 公式索引

每章末尾建议包含公式索引，但不要把复杂公式塞进 Markdown 表格。

推荐格式：

```markdown
### 公式 (X.Y)：公式名称

公式按《公式格式说明.md》的稳定写法书写。

- **含义**：这个公式表达什么。
- **需要掌握到什么程度**：读者应该理解到什么层次。
```

公式索引不只是列表，还要告诉读者掌握程度。

### 9.4 定义索引

每章末尾建议包含定义索引。

推荐格式：

```markdown
| 编号 | 概念 | 一句话含义 |
|---|---|---|
| 定义 X.1 | 行为克隆 | 把专家动作作为监督学习标签训练策略 |
```

定义索引用于帮助读者复习概念，不要写成长篇解释。

## 10. 配图规范

每章建议至少 3 张例图。

这些图不是章节总结图，而是概念讲解图、流程图、对比图、公式推导辅助图或错误模式图。

### 10.1 先盘点并优先复用已有图片

成书级改造章节时，不要默认重新生成所有图片。应先盘点仓库中本章和相关章节已经存在的图片，优先复用仍然准确、风格一致、可解释当前概念的原图。

复用前按四类处理：

```text
A 类：直接复用；
B 类：改名 / 改编号后复用；
C 类：作为补充图复用；
D 类：弃用。
```

图片复用时必须同步检查：

```text
正文图号是否连续；
图片文件名是否容易识别；
图中文字标题是否和正文图号一致；
图注是否说明该图在本章中的作用；
是否存在同一图号对应多张图片的问题。
```

### 10.2 推荐图类型

```text
任务总览图：说明本章讨论的系统对象；
概念关系图：解释状态、观测、动作、轨迹、策略之间的关系；
公式主线图：解释几个核心公式之间如何递进；
推导辅助图：帮助理解命题、证明或等价关系；
流程图：解释训练流程、部署流程、数据闭环；
对比图：解释两种方法、两种分布或两种阶段的差异；
失败模式图：解释为什么算法会翻车。
```

### 10.3 每张图必须回答一个问题

图不是装饰。每张图都要回答一个明确问题。

建议图注格式：

```text
这张图想说明的问题是：……
图中左边表示：……
图中右边表示：……
读者要记住的是：……
```

每章末尾要有两类阅读指引。

### 11.1 建议阅读的附录条目

要说明为什么读，而不是只列附录名。

推荐阅读要说明：

```text
材料类型：基础 / 经典 / 前沿 / 工程；
阅读目的：为什么读；
重点看什么：读哪几节或哪类问题；
对应本章：和本章哪个概念、公式或证明有关。
```

不要只列论文名。

## 14. 一句话原则

每一章最终都要达到：

```text
读者知道本章为什么出现，
知道每个概念怎么定义，
知道每个公式在算什么，
知道核心公式为什么会出现，
知道关键结论为什么成立，
```
