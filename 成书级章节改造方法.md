# 成书级章节改造方法

> 本文档用于指导《模仿学习的数学原理》后续章节的成书级改造。
>
> 第一篇已经完成阶段性改造，本文不再记录第一篇的具体过程，也不再维护某一章的专属模板，而是沉淀为后续全书通用的改造规则。
>
> **公式格式统一由《公式格式说明.md》管理。本文只规定章节结构、写作方法、数学主线、公式推导主线、命题证明、受众定位和成书级检查逻辑。**

---

## 1. 本书的四条顶层约束

后续所有章节改造，都必须同时满足四条约束。

### 1.1 数学对象主线必须推进

本书不是模仿学习方法清单，而是围绕一条数学对象主线展开：

```text
专家轨迹
→ 观测与动作
→ 策略
→ 专家策略与学习策略
→ 训练目标
→ 专家分布与执行分布
→ 分布偏移与误差累积
→ 数据聚合、分布匹配、概率建模、生成式策略
```

每章都要回答：

```text
本章推进了哪一个数学对象？
本章引入了哪一个新的训练目标、分布视角或优化问题？
本章和前后章节的数学关系是什么？
```

如果某一节只是介绍方法现象，但没有推进数学对象、训练目标或数学问题，就需要重写。

### 1.2 公式推导主线必须显性化

《模仿学习的数学原理》不能只解释公式“是什么意思”，还要解释公式“为什么会出现”和“为什么成立”。

每章都应尽量形成一条公式推导主线：

```text
上一章公式留下的问题
→ 本章核心公式从哪里来
→ 本章公式解决了什么问题
→ 本章公式暴露了什么局限
→ 下一章为什么需要新的公式
```

例如第二篇可以形成如下公式主线：

```text
BC 经验风险
→ 最大似然 / 负对数似然
→ 策略诱导轨迹分布 p_pi(tau)
→ 单步 loss 与轨迹目标 J(pi) 不一致
→ 专家分布 d^{pi_E} 与学习策略分布 d^{pi_theta} 不一致
→ MSE 条件均值问题
→ 概率策略 pi(a|s)
→ 隐变量边缘化 p(a|x)=int p(a|x,z)p(z)dz
→ CVAE ELBO
→ ACT / Diffusion Policy 的动作序列分布建模
```

章节中建议保留一个模块：

```markdown
## 本章公式主线
```

这个模块不必很长，但必须回答：

```text
本章核心公式从前一章哪个问题推出来？
它解决了什么问题？
它又留下了什么问题？
```

### 1.3 关键结论必须有推导或证明

本书讲的是“数学原理”，不是只讲“公式解释”。关键结论不能只给出结论，必须尽量给出命题、推导或证明。

不是所有公式都要证明，但以下类型必须优先证明：

```text
会改变工程判断的结论；
连接两个章节主线的结论；
解释某个方法为什么失败的结论；
说明某个训练目标为什么等价于另一个目标的结论；
引出后续方法的关键数学结论。
```

优先补充证明的典型结论包括：

```text
最大似然等价于最小化负对数似然；
固定方差高斯负对数似然等价于 MSE；
MSE 的最优预测是条件均值；
BC 的误差累积风险；
DAgger 修正的是访问状态分布而不是简单换 loss；
CVAE 的 ELBO 推导；
KL 项约束 posterior 与 prior 的关系；
Diffusion 去噪目标与 score / 条件动作分布的关系；
Flow Matching 速度场目标的基本推导。
```

证明必须服务于读者理解和工程判断，不是为了炫技。

### 1.4 受众是普通工程师

本书主要写给有工程背景、数学基础一般、希望理解模仿学习本质的算法工程师和学习者。

默认读者具备：

```text
基本机器学习概念；
基本监督学习、loss、训练集、验证集概念；
一点 PyTorch / 算法工程经验；
但不默认熟悉概率统计、强化学习理论、机器人学习理论。
```

因此章节写作必须做到：

```text
公式出现前先说明工程问题；
公式出现后逐项解释符号；
关键结论要有推导或证明；
证明不能跳步到“显然”“可得”“标准结论”；
抽象概念必须落回机械臂、机器人或策略学习场景；
不要用高级理论术语替代解释；
不默认读者熟悉概率统计、强化学习或机器人学习背景。
```

### 1.5 读完必须能形成工程判断

每章不仅要让读者“看懂”，还要让读者“能判断”。

例如读完某章后，读者应该能判断：

```text
BC 验证集 loss 很低但闭环失败，可能是训练分布和执行分布不一致；
MSE 在多模态动作上失败，可能是动作分布被平均；
DAgger 不是换损失函数，而是在修正学习策略访问分布的数据覆盖问题；
Diffusion Policy 不是简单换网络，而是在建模条件动作分布或动作序列分布。
```

后续章节建议保留一个模块：

```markdown
## 读完本章，你应该能判断什么？
```

这个模块用于把公式、证明、概念和工程决策显性连接起来，避免章节变成“知识正确但读者不知道有什么用”。

## 2. 成书级章节改造目标

成书级章节改造不是简单润色，也不是扩写字数，而是让每一章具备以下能力：

1. **概念递进清楚**：先定义基础对象，再引入依赖它们的复杂对象。
2. **数学符号清楚**：第一次出现的变量、公式、条件概率、损失函数都要解释。
3. **公式主线清楚**：每章要说明核心公式从哪里来、解决什么问题、留下什么问题。
4. **关键证明清楚**：关键结论要有命题、推导或证明，不能只写结论。
5. **工程含义清楚**：每个核心公式和关键证明都要能落回机械臂抓取与放置主线任务中理解。
6. **章节结构稳定**：章标题、小节标题、公式主线、公式索引、定义索引、思考题、推荐阅读形成统一格式。
7. **网页显示稳定**：公式、图片、表格、列表都要适合 mdBook 和 GitHub 网页阅读。
8. **读者路径友好**：面向普通工程师，不假设读者已经熟悉概率统计、强化学习或机器人学习背景。

一句话概括：

```text
先让读者知道问题从哪里来，
再定义概念，
再写公式，
再说明公式如何推出来，
再证明关键结论，
再落到工程任务，
最后指出局限和下一章问题。
```

## 3. 章节改造的基本方法

### 3.1 从真实困惑进入

章节开头不能只说“本章介绍某方法”，而要回答读者为什么需要这一章。

推荐开篇回答：

```text
上一章留下了什么问题？
工程实践中会遇到什么困惑？
如果不懂本章，会在哪些判断上出错？
```

例如：

```text
如果我们已经有专家示范，最直接的方法是不是把观测当输入、专家动作当标签做监督学习？
这个看似自然的想法为什么会在闭环执行中翻车？
```

### 3.2 按概念依赖链组织小节

章节小节顺序必须遵守概念依赖关系，避免“公式里已经出现的对象尚未定义”。

推荐链路：

```text
上一章问题
→ 本章真实困惑
→ 基础对象定义
→ 新数学对象
→ 本章公式主线
→ 训练目标或优化目标
→ 关键命题 / 推导 / 证明
→ 公式拆解
→ 工程解释
→ 失败模式或局限
→ 下一章问题
```

### 3.3 第一次出现的重要概念必须正式定义

凡是第一次出现、后续会反复使用的概念，都应该用“定义 X.Y”显式给出。

定义格式：

```markdown
> **定义 X.Y：概念名**
>
> 用一段简洁准确的话定义该概念。
```

定义之后必须接解释：

```text
这个定义里有哪几个关键词？
这些关键词分别对应什么工程对象？
为什么后续章节会反复用到它？
```

定义不宜过长。定义负责“准确”，后面的段落负责“解释”。

### 3.4 核心公式必须讲清楚

核心公式不能直接抛给读者，必须按下面顺序拆解：

```text
动机
→ 公式
→ 读法
→ 符号解释
→ 直觉
→ 工程含义
→ 常见误解
```

所有重要公式都要尽量遵守这个结构，尤其是：

```text
策略公式；
损失函数；
概率分布；
最大似然；
负对数似然；
期望；
KL 散度；
occupancy measure；
diffusion / flow matching 相关公式。
```

公式格式、GitHub 网页兼容性、mdBook 兼容性全部以：

```text
公式格式说明.md
```

为唯一准则。

### 3.5 每章必须有公式主线

公式主线不是简单列公式，而是说明公式之间的推进关系。

推荐格式：

```markdown
## 本章公式主线

上一章已经得到：……
但它留下的问题是：……
因此本章引入：……
这个公式说明：……
它解决了：……
但它仍然不能解决：……
所以下一章需要：……
```

例如第7章可以写成：

```text
第6章告诉我们闭环目标是 J(pi)=E_{tau~p_pi}[C(tau)]，
但单步训练仍常用 MSE。
MSE 在同一状态多动作模式下的最优解是条件均值。
条件均值可能不是任何一个可执行动作。
所以需要把策略从 a=f(s) 推进到 pi(a|s)。
```

这类模块用于增强“公式推进感”，避免章节只是公式解释的堆叠。

### 3.6 关键结论必须写成命题、推论或证明

关键结论建议使用“命题 / 推论 / 证明”结构。

推荐格式：

```markdown
> **命题 X.Y：结论名称**
>
> 在什么假设下，什么结论成立。

**证明思路**：

先用一段话解释为什么要证明它，以及证明将如何展开。

**证明**：

逐步推导，不跳过对普通工程师不显然的步骤。

**这个命题告诉我们什么？**

把数学结论翻译成工程判断。

**常见误解**：

指出读者容易误读的地方。
```

如果证明很长，可以在正文给核心推导，把完整证明放入附录，并在正文中说明：

```text
完整证明见附录 X。
```

### 3.7 证明要分层，不要把正文写成纯数学论文

证明建议分三层：

```text
第一层：直觉推导
第二层：正文中的核心证明
第三层：附录中的完整证明或补充
```

正文中优先证明那些能直接改变工程判断的结论。

例如：

```text
MSE 最优预测是条件均值；
固定方差高斯 NLL 等价于 MSE；
DAgger 修正的是访问状态分布；
CVAE ELBO 如何从边缘 likelihood 推出来。
```

过于基础或过长的概率推导可以放到附录，但正文必须告诉读者它为什么重要。

### 3.8 每章都要落回工程主线

本书采用“视觉引导机械臂抓取与放置任务”作为统一工程主线。

每一章都要尽量回答：

```text
本章的数学概念，在机械臂抓取与放置任务中解决了什么问题？
如果这个数学假设不成立，工程上会出现什么失败？
这个公式或证明能帮助工程师做什么判断？
```

例如，讲 BC 损失时不能只解释成“监督学习误差”，还要说明：

```text
在机械臂任务中，这个损失表示：模型看到当前图像和机械臂状态后，输出的末端控制命令要尽量接近专家遥操作命令。
```

讲 MSE 条件均值证明时，也不能只停在求导，而要说明：

```text
如果左抓和右抓都是专家成功动作，MSE 学到的条件均值可能落在两种抓法中间，变成不可执行动作。
```

## 4. 推荐章节结构

不是每章都必须完全一致，但核心模块建议保留。

```text
第X章 标题：本章知识推进方向

本章一句话导读

X.1 从上一章问题或工程失败出发
X.2 本章要解决的核心问题
X.3 本章在全书数学主线中的位置
X.4 定义本章核心概念
X.5 本章公式主线
X.6 引入本章核心数学对象
X.7 推导或解释核心训练目标
X.8 关键命题、推论或证明
X.9 拆解核心公式
X.10 放回机械臂任务理解
X.11 工程实现中的注意事项
X.12 方法局限与下一章问题
X.13 本章总结
X.14 读完本章，你应该能判断什么
X.15 本章公式索引
X.16 本章定义索引
X.17 建议阅读的附录条目
X.18 思考题
X.19 本章配图清单
X.20 推荐阅读与深入材料
```

可根据章节性质调整，但不要丢失四类内容：

```text
数学对象主线推进；
公式推导主线推进；
关键结论有推导或证明；
普通工程师能形成工程判断。
```

## 5. 标题与小节标题规则

### 5.1 章标题要表达知识推进路径

标题最好包含两个部分：

```text
主题对象 + 本章推进方向
```

例如：

```text
Behavior Cloning：从专家样本到监督学习策略
分布偏移：为什么训练误差小不代表执行成功
DAgger：用交互数据修正策略分布
CVAE：从隐变量动作模式到可训练生成策略
```

### 5.2 小节标题要专业但可读

推荐标题风格：

```text
概念名：本节回答的问题
```

例如：

```text
观测与状态：策略输入从哪里来
动作与动作空间：策略输出到哪里去
轨迹：专家行为的时间结构
公式主线：为什么从单步 loss 推到轨迹目标
命题：MSE 为什么会学到条件均值
```

不建议大量使用：

```text
第一层问题：……
第二层问题：……
第三层问题：……
```

这种写法适合草稿推理，但成书目录显得不够正式。

## 6. 公式主线、证明、公式索引与定义索引

### 6.1 公式主线

每章建议包含“本章公式主线”模块。

推荐格式：

```markdown
## 本章公式主线

上一章的核心公式是：……
它说明了：……
但它留下了：……
本章因此引入：……
本章核心公式说明：……
它为下一章铺垫：……
```

公式主线不是公式索引。公式索引回答“本章有哪些公式”，公式主线回答“这些公式为什么按这个顺序出现”。

### 6.2 命题与证明

关键结论建议写成命题、推论或证明。

推荐格式：

```markdown
> **命题 X.Y：结论名称**
>
> 在什么假设下，什么结论成立。

**证明思路**：……

**证明**：……

**这个命题告诉我们什么？**：……

**常见误解**：……
```

注意：证明中的公式格式仍然必须遵守《公式格式说明.md》。不要因为证明较长，就使用 GitHub 不稳定的公式写法。

### 6.3 公式索引

每章末尾建议包含公式索引，但不要把复杂公式塞进 Markdown 表格。

推荐格式：

```markdown
### 公式 (X.Y)：公式名称

公式按《公式格式说明.md》的稳定写法书写。

- **含义**：这个公式表达什么。
- **需要掌握到什么程度**：读者应该理解到什么层次。
```

公式索引不只是列表，还要告诉读者掌握程度。

### 6.4 定义索引

每章末尾建议包含定义索引。

推荐格式：

```markdown
| 编号 | 概念 | 一句话含义 |
|---|---|---|
| 定义 X.1 | 行为克隆 | 把专家动作作为监督学习标签训练策略 |
```

定义索引用于帮助读者复习概念，不要写成长篇解释。

## 7. 配图规范

每章建议至少 3 张例图。

这些图不是章节总结图，而是概念讲解图、流程图、对比图、公式推导辅助图或错误模式图。

### 7.1 先盘点并优先复用已有图片

成书级改造章节时，不要默认重新生成所有图片。应先盘点仓库中本章和相关章节已经存在的图片，优先复用仍然准确、风格一致、可解释当前概念的原图。

复用前按四类处理：

```text
A 类：直接复用；
B 类：改名 / 改编号后复用；
C 类：作为补充图复用；
D 类：弃用。
```

图片复用时必须同步检查：

```text
正文图号是否连续；
图片文件名是否容易识别；
图中文字标题是否和正文图号一致；
图注是否说明该图在本章中的作用；
是否存在同一图号对应多张图片的问题。
```

### 7.2 推荐图类型

```text
任务总览图：说明本章讨论的系统对象；
概念关系图：解释状态、观测、动作、轨迹、策略之间的关系；
公式主线图：解释几个核心公式之间如何递进；
推导辅助图：帮助理解命题、证明或等价关系；
流程图：解释训练流程、部署流程、数据闭环；
对比图：解释两种方法、两种分布或两种阶段的差异；
失败模式图：解释为什么算法会翻车。
```

## 8. 推荐阅读与附录联动

每章末尾要有两类阅读指引。

### 8.1 建议阅读的附录条目

要说明为什么读，而不是只列附录名。

例如：

```text
附录 B：如果不熟悉条件概率和最大似然，用它补足概率基础。
附录 D：如果不理解 MSE 与高斯 NLL 的关系，用它补足连续动作回归基础。
附录 G：如果不熟悉 VAE / CVAE，用它补足生成模型基础。
```

如果正文证明依赖某个基础工具，也要指向对应附录。

例如：

```text
ELBO 推导中用到 Jensen 不等式，如果不熟悉，请阅读附录 B 或附录 G 中的相关条目。
```

### 8.2 推荐阅读与深入材料

推荐阅读要说明：

```text
材料类型：基础 / 经典 / 前沿 / 工程；
阅读目的：为什么读；
重点看什么：读哪几节或哪类问题；
对应本章：和本章哪个概念、公式或证明有关。
```

不要只列论文名。

## 9. 成书级检查清单

每章完成后，用以下清单检查。

### 9.1 顶层定位检查

- [ ] 本章是否推进了全书数学对象主线，而不是只罗列方法？
- [ ] 本章是否显式推进了公式推导主线？
- [ ] 本章是否默认读者是普通工程师，并避免跳过必要背景？
- [ ] 本章是否解释了公式背后的工程问题？

### 9.2 结构检查

- [ ] 章标题是否表达了本章知识推进路径？
- [ ] 小节顺序是否符合概念依赖关系？

- [ ] 第一次出现的重要概念是否正式定义？
- [ ] 定义是否简洁、准确、不过度展开？
- [ ] 定义之后是否有直觉解释和机械臂例子？
- [ ] 是否有本章定义索引？

### 9.4 公式检查

- [ ] 是否严格遵守《公式格式说明.md》？

- [ ] 每个核心公式是否有符号解释？
- [ ] 每个核心公式是否有工程含义？
- [ ] 是否说明核心公式从哪里来、解决什么问题？

### 9.5 推导与证明检查

- [ ] 关键结论是否写成命题、推论或证明？
- [ ] 证明是否说明了假设条件？
- [ ] 证明是否避免跳步到“显然”“可得”？
- [ ] 证明之后是否解释了工程含义？
- [ ] 如果证明较长，是否把完整证明或基础工具引导到附录？
- [ ] 是否优先证明了会改变工程判断的结论？

### 9.6 工程主线检查

- [ ] 本章是否回到机械臂抓取与放置任务？
- [ ] 是否解释了本章概念解决哪类真实失败？

- [ ] 是否指出了工程实现风险？
- [ ] 是否说明了关键证明能帮助工程师做什么判断？

### 9.7 图示与配套检查

- [ ] 是否优先复用仍然准确的原图？
- [ ] 本章是否至少有 3 张例图？
- [ ] 图是否服务于概念理解、公式推导或失败解释，而不是简单总结？

## 11. 一句话原则

```text
读者知道本章为什么出现，
```
