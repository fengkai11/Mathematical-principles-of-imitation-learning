# 成书级章节改造方法

> 本文档用于指导《模仿学习的数学原理》后续章节的系统性改造。它总结第 1–4 章从“普通解释稿”改造成“成书级章节”的方法，后续章节都可以按此规范检查和重写。
>
> **重要约定**：本文只规定章节写作方法和成书级检查逻辑；所有公式格式、公式渲染兼容性、GitHub / mdBook 显示规则，统一以仓库根目录的《公式格式说明.md》为准。本文不再重复维护公式格式细则，避免规则分散和相互冲突。

---

## 1. 改造目标

成书级章节改造不是简单润色，也不是扩写字数，而是让每一章都具备以下能力：

1. **概念递进清楚**：先定义基础对象，再引入依赖它们的复杂对象。
2. **数学符号清楚**：第一次出现的变量、公式、条件概率、损失函数都要解释。
3. **工程含义清楚**：每个公式都要能落回机械臂抓取与放置主线任务中理解。
4. **章节结构稳定**：章标题、小节标题、公式索引、定义索引、思考题、推荐阅读形成统一格式。
5. **网页显示稳定**：公式、图片、表格、列表都要适合 mdBook 和 GitHub 网页阅读。
6. **读者路径友好**：面向只有基础数学知识的算法工程师，不假设读者已经熟悉概率统计、强化学习或机器人学习背景。

一句话概括：

```text
先让读者知道问题从哪里来，再定义概念，再写公式，再拆公式，再落到工程任务，最后指出局限和下一章问题。
```

### 1.1 三条顶层约束

后续每一章都必须同时满足三条顶层约束。

#### 约束一：数学主线必须推进

本书不是模仿学习方法清单，而是围绕一条数学主线展开：

```text
专家轨迹
→ 观测与动作
→ 策略
→ 专家策略与学习策略
→ 训练目标
→ 专家分布与执行分布
→ 分布偏移与误差累积
→ 数据聚合、分布匹配、概率建模、生成式策略
```

每章都要明确回答：

```text
本章推进了哪一个数学对象？
本章引入了哪一个新的训练目标或分布视角？
本章和前后章节的数学关系是什么？
```

如果某一节只是介绍方法现象，但没有推进数学对象、数学目标或数学问题，就需要重写。

#### 约束二：受众是普通工程师

本书主要写给有工程背景、数学基础一般、希望理解模仿学习本质的算法工程师和学习者。

默认读者具备：

```text
基本机器学习概念；
基本监督学习、loss、训练集、验证集概念；
一点 PyTorch / 算法工程经验；
但不默认熟悉概率统计、强化学习理论、机器人学习理论。
```

因此，章节不能只追求理论完整，还必须做到：

```text
公式出现前先说明工程问题；
公式出现后逐项解释符号；
抽象概念必须落回机械臂、机器人或策略学习场景；
不要用高级理论术语替代解释；
不要假设读者知道“显然”“可得”“标准结论”。
```

#### 约束三：读完必须能形成工程判断

每章不仅要让读者“看懂”，还要让读者“能判断”。

每章至少要让读者获得一种判断能力，例如：

```text
看到 BC 验证集 loss 很低但闭环失败时，能判断可能是训练分布和执行分布不一致；
看到 MSE 在多模态动作上失败时，能判断可能是动作分布被平均；
看到 DAgger 时，能判断它是在修正学习策略访问分布的数据覆盖问题；
看到 Diffusion Policy 时，能判断它不是简单换网络，而是在建模条件动作分布或动作序列分布。
```

后续章节建议增加或保留一个模块：

```markdown
## 读完本章，你应该能判断什么？
```

这个模块用于把公式、概念和工程决策显性连接起来，避免章节变成“知识正确但读者不知道有什么用”。

## 2. 总体改造原则

### 2.1 从“问题罗列”改为“概念依赖链”

章节小节顺序必须遵守概念依赖关系。

例如第 1 章最终采用的顺序是：

```text
任务
→ 观测与状态
→ 动作与动作空间
→ 观测—动作样本
→ 轨迹
→ 示范数据集
→ 策略
→ 专家策略
→ 闭环执行
```

这样可以避免出现下面的问题：

```text
轨迹公式里已经出现 o_t 和 a_t，
但观测和动作还没有正式定义。
```

后续章节同理。比如第 2 章 Behavior Cloning 应该遵守：

```text
第一章留下的问题
→ 行为克隆定义
→ 训练样本
→ 策略模型形式
→ 损失函数
→ 经验风险最小化
→ 机械臂训练流程
→ 闭环执行风险
→ 本章总结
```

### 2.2 第一次出现的重要概念必须正式定义

不能只用描述性文字带过。凡是第一次出现、后续会反复使用的概念，都应该用“定义 X.X”显式给出。

定义格式建议：

```markdown
> **定义 2.1：行为克隆**
>
> 行为克隆（Behavior Cloning, BC）是指把专家示范中的观测—动作样本看作监督学习数据，训练一个策略模型，使其在给定观测时输出接近专家动作的动作。
```

定义之后必须接解释：

```text
这个定义里有三个关键词：观测—动作样本、监督学习、策略模型。
```

定义不宜过长。定义负责“准确”，后面的段落负责“解释”。

### 2.3 公式必须按“动机—公式—读法—符号—直觉—工程含义”拆解

核心公式不能直接抛给读者。

推荐结构：

```markdown
**公式 (2.4)：连续动作下的行为克隆 MSE 损失**

公式按《公式格式说明.md》的稳定写法书写。

这个公式读作：……

其中：

- $\theta$：……
- $N$：……
- $o_i$：……
- $a_i$：……
- $f_\theta(o_i)$：……
- $\ell(\cdot, \cdot)$：……

直觉上，它是在做一件事：……

放到机械臂任务中，它表示：……

常见误解是：……
```

所有重要公式都要尽量遵守这个结构，尤其是：

- 策略公式；
- 损失函数；
- 概率分布；
- 最大似然；
- 负对数似然；
- 期望；
- KL 散度；
- occupancy measure；
- diffusion / flow matching 相关公式。

### 2.4 每章必须从真实困惑进入

章节开头不能只说“本章介绍某方法”，而要回答读者为什么需要这一章。

推荐开篇方式：

```text
上一章留下了什么问题？
工程实践中会遇到什么困惑？
如果不懂本章，会在哪些判断上出错？
```

例如第 2 章不能只说“介绍 Behavior Cloning”，而要强调：

```text
如果我们已经有专家示范，最直接的方法是不是把观测当输入、专家动作当标签做监督学习？
这个看似自然的想法为什么会在闭环执行中翻车？
```

## 3. 公式格式规范入口

公式格式不在本文中重复维护。后续所有章节改造、公式校对、公式索引整理，必须优先遵守仓库根目录：

```text
公式格式说明.md
```

该文档统一管理以下内容：

```text
1. 行内公式和独立公式的写法；
2. GitHub 网页与 mdBook 的双重渲染兼容性；
3. 等号不能单独占一行；
4. 不使用 GitHub 禁用宏，例如 \operatorname / \operatorname*；
5. argmin 使用 \arg\min；
6. 向量范数使用 \lVert ... \rVert；
7. 不把块级公式放进 blockquote；
8. 不把复杂公式塞进 Markdown 表格；
9. 公式索引与正文公式保持一致；
10. 公式提交前的检查清单。
```

本文只保留公式讲解要求：

> **公式不只是要显示正确，还要讲清楚为什么出现、怎么读、每个符号是什么意思、在机械臂任务中对应什么工程含义。**

## 4. 章节标题与小节标题改造方法

### 4.1 章标题要表达本章知识推进路径

不建议只用口语化标题，例如：

```text
模仿学习到底在模仿什么
```

可以改成：

```text
第1章 从专家示范到策略学习：模仿学习问题的起点
```

标题最好包含两个部分：

```text
主题对象 + 本章推进方向
```

例如：

```text
第2章 Behavior Cloning：从专家样本到监督学习策略
第3章 分布偏移：为什么训练误差小不代表执行成功
第4章 DAgger：用交互数据修正策略分布
```

### 4.2 小节标题要专业但可读

推荐标题风格：

```text
概念名：本节回答的问题
```

例如：

```text
1.2 观测与状态：策略输入从哪里来
1.3 动作与动作空间：策略输出到哪里去
1.4 观测—动作样本：最小监督学习单元
1.5 轨迹：专家行为的时间结构
```

不建议大量使用：

```text
第一层问题：……
第二层问题：……
第三层问题：……
```

这种写法适合草稿推理，但成书目录显得不够正式。

## 5. 机械臂主线绑定方法

本书采用“视觉引导机械臂抓取与放置任务”作为统一主线。

每一章都要尽量回答：

```text
本章的数学概念，在机械臂抓取与放置任务中解决了什么问题？
```

### 5.1 讲概念时绑定机械臂

示例：

```text
观测 o_t：顶部图像、侧视图像、关节角、末端位姿、夹爪状态。
动作 a_t：末端位置增量、末端姿态增量、夹爪开合命令。
轨迹 tau：一次完整抓取与放置示范过程。
策略 pi_theta：根据当前观测输出下一步动作的规则。
```

### 5.2 讲公式时绑定机械臂

例如 Behavior Cloning 的损失函数不能只解释成“监督学习误差”，还要写成：

```text
在机械臂任务中，这个损失表示：模型看到当前图像和机械臂状态后，输出的末端控制命令要尽量接近专家遥操作命令。
```

### 5.3 讲局限时绑定机械臂

比如 BC 的局限不能只说“分布偏移”，还要说：

```text
如果模型第一步让夹爪偏了 2 厘米，下一帧相机看到的画面就不再是专家数据中的画面。模型继续按照专家数据训练出的局部规则行动，可能越走越偏，最后夹住空气或撞到物体边缘。
```

## 6. 配图规范

每章至少 3 张例图。

注意：这些图不是章节总结图，而是概念讲解图、流程图、对比图或错误模式图。

### 6.1 先盘点并优先复用已有图片

成书级改造章节时，不要默认重新生成所有图片。应先盘点仓库中本章和相关章节已经存在的图片，优先复用仍然准确、风格一致、可解释当前概念的原图。

复用前按四类处理：

```text
A 类：直接复用
图的内容仍然准确，图号和正文位置也基本匹配，只需要在正文中重新引用。

B 类：改名 / 改编号后复用
图内容可用，但文件名、图号、标题或正文位置和新版章节结构不一致，需要同步修改图注或正文引用。

C 类：作为补充图复用
新版章节已有主图，旧图可以作为局部概念补充图，例如补充“监督学习视角”“训练流程细节”“错误模式”。

D 类：弃用
图风格、编号、内容或概念表达已经和新版章节不一致，保留在仓库中，但不再正文引用。
```

图片复用时必须同步检查：

- 正文图号是否连续；
- 图片文件名是否容易识别；
- 图中文字标题是否和正文图号一致；
- 图注是否说明该图在本章中的作用；
- 是否存在同一图号对应多张图片的问题。

### 6.2 推荐图类型

1. **任务总览图**：用于告诉读者本章讨论的系统对象是什么。
2. **概念关系图**：用于解释状态、观测、动作、轨迹、策略之间的关系。
3. **流程图**：用于解释训练流程、部署流程、数据闭环。
4. **对比图**：用于解释确定性策略 vs 随机策略、训练阶段 vs 执行阶段、专家分布 vs 学习策略分布。
5. **失败模式图**：用于解释为什么算法会翻车。

## 9. 定义索引与公式索引要求

### 9.1 定义索引

每章末尾建议包含：

```markdown
## X.13 本章定义索引

| 编号 | 概念 | 一句话含义 |
|---|---|---|
| 定义 X.1 | 行为克隆 | 把专家动作作为监督学习标签训练策略 |
```

### 9.2 公式索引

每章末尾建议包含公式索引，但公式格式必须引用《公式格式说明.md》，不要在本文中重复维护。

推荐格式：

```markdown
## X.12 本章公式索引

### 公式 (X.1)：公式名称

公式按《公式格式说明.md》的稳定写法书写。

- **含义**：确定性策略输入观测，输出一个预测动作。
- **需要掌握到什么程度**：理解确定性策略如何输出动作。
```

公式索引不只是列表，还要告诉读者掌握程度。

## 10. 推荐阅读与附录联动

每章末尾要有两类阅读指引。

### 10.1 建议阅读的附录条目

例如：

```text
附录 A：数学符号与公式阅读方法
附录 B：概率论最小生存包
附录 C：优化与梯度下降基础
附录 F：强化学习与序列决策基础
```

要说明为什么读：

```text
如果读者不熟悉负对数似然，需要阅读附录 B 中的条件概率和最大似然部分。
```

### 10.2 推荐阅读与深入材料

推荐阅读要说明：

```text
材料类型：基础 / 经典 / 前沿 / 工程
阅读目的：为什么读
重点看什么：读哪几节或哪类问题
对应本章：和本章哪个概念有关
```

不要只列论文名。

## 11. 成书级检查清单

每章完成后，用以下清单检查。

### 11.0 顶层定位检查

- [ ] 本章是否推进了全书数学主线，而不是只罗列方法？
- [ ] 本章是否默认读者是普通工程师，并避免跳过必要背景？
- [ ] 本章是否解释了公式背后的工程问题？
- [ ] 本章是否让读者看完后能形成至少一个工程判断？
- [ ] 是否有“读完本章，你应该能判断什么”或等价内容？

### 11.1 结构检查

- [ ] 章标题是否表达了本章知识推进路径？
- [ ] 小节顺序是否符合概念依赖关系？
- [ ] 是否存在“后定义的概念先出现在公式中”的问题？
- [ ] 是否和上一章、下一章形成自然衔接？

### 11.2 概念检查

- [ ] 第一次出现的重要概念是否正式定义？
- [ ] 定义是否简洁、准确、不过度展开？

### 11.3 公式检查

### 11.4 工程主线检查

- [ ] 是否解释了本章概念解决哪类真实失败？

## 12. 给 GPT / Codex 的改造提示词模板

后续改造章节时，可以使用以下提示词：

```text
请按照仓库中的《成书级章节改造方法.md》和《公式格式说明.md》，对第 X 章进行成书级改造。

要求：
1. 不改变全书总体目录和本章核心主题；
2. 本章必须推进“模仿学习背后的数学主线”，不要只罗列方法现象；
3. 默认读者是有工程背景、数学基础一般的普通算法工程师，不能跳过必要背景解释；
4. 本章必须让读者读完后能形成工程判断，例如能解释某类失败、选择某类方法或判断某个训练目标的局限；
5. 先检查小节顺序是否符合概念依赖关系，如不合理请调整；
6. 第一次出现的重要概念必须给出“定义 X.Y”；
7. 核心公式必须按“动机—公式—读法—符号解释—直觉—工程含义—常见误解”拆解；
8. 公式格式、GitHub 网页兼容性、mdBook 兼容性全部以《公式格式说明.md》为准，不要自行发明公式写法；
9. 公式索引不要把复杂公式塞进 Markdown 表格，优先使用逐条公式索引，并确保索引公式与正文一致；
10. 改造章节前先盘点已有图片，优先复用准确可用的原图，再决定是否新增图片；
11. 所有概念尽量落回“视觉引导机械臂抓取与放置任务”解释；
12. 每章至少 3 张概念讲解图，不要章节总结图；
13. 保留或补充“读完本章，你应该能判断什么”、本章公式索引、定义索引、要点回顾、附录阅读、思考题、配图清单和推荐阅读；
14. 删除内部更新说明、写作过程说明、mdBook 兼容说明等非正文内容；
15. 输出成书级正文，不要只给修改建议。
```

## 13. 一句话原则

每一章最终都要达到：

```text
读者知道本章为什么出现，
知道每个概念怎么定义，
知道每个公式在算什么，
知道它在机械臂任务中对应什么，
知道它为什么还不够、下一章要解决什么，
并且知道读完本章后自己能做出什么工程判断。
```
