# 第12章：Offline Imitation Learning：离线数据不是越多越好，是坑有没有录进去

> **新版布局位置**：本章属于 **第三篇：经典模仿学习的分布匹配与奖励视角**。

> **本章一句话导读**：第11章 GAIL 说明了在线 rollout 可以帮助匹配专家与策略的 occupancy measure，但真实机器人不能随便试错。本章进入 Offline Imitation Learning：当训练主要依赖固定数据集时，核心风险不是数据量不够大，而是 behavior policy 产生的数据是否覆盖了部署策略会遇到的状态、动作、失败和恢复。

---

## 1. 从第11章留下的问题出发

第11章 GAIL 的核心思想是：

```text
让当前策略 rollout
→ 得到当前策略的状态—动作样本
→ 用判别器比较专家样本和策略样本
→ 用判别器反馈更新策略
→ 让策略的 occupancy measure 越来越像专家
```

这条路线在数学上很漂亮，因为它直接关注策略闭环执行后会访问哪些状态、执行哪些动作。

但工程上马上会遇到一个现实问题：

> 如果当前策略还很差，我们敢让它在真实机器人上反复 rollout 吗？

在仿真里，策略撞墙可以重置；在游戏里，角色失败可以重新开始；在离线实验里，失败最多损失一点计算时间。

真实机器人不是这样。

机械臂抓取任务中，一次错误 rollout 可能导致：

- 夹具撞弯；
- 工件划伤；
- 治具卡死；
- 相机支架抖动导致标定漂移；
- 现场节拍被打断；
- 工程师需要人工恢复设备；
- 安全员直接禁止继续探索。

所以，第12章要处理的是第11章之后自然出现的问题：

> 如果我们不能频繁在线 rollout，只能拿已有数据训练策略，那么模仿学习的核心风险会变成什么？

答案是：离线数据覆盖、support mismatch、OOD action 和 recovery data。

本章标题说“离线数据不是越多越好，是坑有没有录进去”，不是玩笑，而是离线模仿学习的工程真相。

![图12-1 数据覆盖度示意图](../images/图12-1_数据覆盖度示意图.png)

**图12-1 说明**：

- 绿色和蓝色区域表示离线数据覆盖较多的常见状态—动作组合；
- 红色空洞表示失败、恢复、危险边界、罕见姿态等缺口；
- 训练 loss 主要由数据覆盖区域决定；
- 部署风险往往来自这些没有覆盖或覆盖很少的区域；
- 所以离线数据质量不能只看数量，而要看任务风险是否被覆盖。

## 3. 本章新增数学对象

| 数学对象 | 符号 | 一句话含义 | 工程对应物 | 后续作用 |
|---|---|---|---|---|
| 离线数据集 | $\mathcal{D}$ | 训练前已经采好的固定轨迹集合 | 遥操作日志、路采数据、仿真录屏 | 离线训练基础 |
| 行为策略 | $\beta(a \mid s)$ | 产生离线数据的策略 | 人类遥操作员、旧模型、规则系统、混合采集流程 | 决定数据分布 |
| 数据分布 | $\rho_\beta(s,a)$ | behavior policy 闭环执行留下的状态—动作分布 | 历史数据覆盖区域 | 判断数据覆盖 |
| 部署策略分布 | $\rho_{\pi_\theta}(s,a)$ | 当前模型上线后诱导的状态—动作分布 | 模型实际会走到哪里、做什么 | 判断部署风险 |
| 支撑集 | $support(\mathcal{D})$ | 数据中实际覆盖到的区域 | 见过的正常、边界、失败、恢复场景 | 分析 OOD |
| support mismatch | - | 训练数据覆盖和部署策略访问区域不一致 | 训练没见过，上线却遇到了 | 离线失败核心原因 |
| OOD action | - | 超出数据支撑集的动作 | 奇怪末端增量、过大速度、异常夹爪命令 | 引发后续 OOD state |
| recovery data | - | 从偏差、失败或边界状态恢复到正常任务的轨迹 | 纠偏、急停后恢复、重新对齐、重新抓取 | 提升闭环鲁棒性 |

## 5. 什么是 Offline Imitation Learning

Offline Imitation Learning 可以翻译为“离线模仿学习”。它的基本设定是：

> 给定一批已经采好的轨迹数据，训练一个策略；训练过程中不允许或很少允许当前策略与环境交互。

数学上，离线数据集可以写成：

$$\mathcal{D} = \{\tau_i\}_{i=1}^{N} \tag{12.1}$$

每条轨迹是：

$$\tau_i = (s_0^i,a_0^i,s_1^i,a_1^i,\dots,s_T^i,a_T^i) \tag{12.2}$$

如果是视觉机器人任务，也常写成观测形式：

$$\tau_i = (o_0^i,a_0^i,o_1^i,a_1^i,\dots,o_T^i,a_T^i) \tag{12.3}$$

这里 $o_t$ 可以包含图像、深度图、关节状态、末端位姿、力传感器读数和语言指令。

离线模仿学习训练一个策略：

$$\pi_\theta(a \mid s) \quad \text{或} \quad \pi_\theta(a \mid o) \tag{12.4}$$

训练时，策略不能像 GAIL 那样频繁到环境里 rollout。它只能从 $\mathcal{D}$ 中学习。

这就是离线学习的基本矛盾：

```text
训练时看到的是固定数据集 D；
部署时面对的是策略自己诱导出来的新分布。
```

## 6. behavior policy：数据不是从天上掉下来的

很多人谈离线数据时，只说“我有多少条轨迹”。

但更重要的问题是：

> 这些轨迹是谁采出来的？

产生离线数据的策略叫 behavior policy，记作：

$$\beta(a \mid s) \tag{12.5}$$

它可以是：

- 人类遥操作员；
- 一个旧版本模型；
- 规则控制器；
- 专家和新手混合数据；
- 安全员接管后的恢复策略；
- 仿真中加入噪声或扰动的数据采集策略。

behavior policy 决定了离线数据会覆盖哪些状态、动作和失败模式。

如果 $\beta$ 只在理想条件下采集成功轨迹，那么数据就主要覆盖顺利执行区域。

如果 $\beta$ 包含扰动、失败、接管和恢复，那么数据会覆盖更多边界状态。

所以，离线数据不是客观世界的完整百科全书。它是 behavior policy 在特定设备、场景、任务和安全约束下留下的历史痕迹。

### 6.1 数据分布由 behavior policy 诱导

类似第11章的 occupancy measure，behavior policy 也会诱导一个状态—动作分布：

$$\rho_\beta(s,a) \tag{12.6}$$

这个分布表示：采集数据时，系统有多经常访问状态 $s$ 并执行动作 $a$。

离线数据 $\mathcal{D}$ 可以看成从 $\rho_\beta(s,a)$ 中采样得到的有限样本。

而模型部署后的分布是：

$$\rho_{\pi_\theta}(s,a) \tag{12.7}$$

Offline IL 的核心风险就是：

$$\rho_{\pi_\theta}(s,a) \neq \rho_\beta(s,a) \tag{12.8}$$

这句话的工程含义是：

> 训练数据记录的是过去某个采集策略会遇到什么；部署模型会遇到什么，则由当前模型和环境共同决定。

### 命题 12.1：离线数据多，不等于闭环部署安全

> **命题 12.1：数据量不能替代分布覆盖**
>
> 如果训练数据主要来自 behavior policy $\beta$ 的分布 $\rho_\beta$，而部署策略 $\pi_\theta$ 诱导出的分布 $\rho_{\pi_\theta}$ 与其不一致，那么即使离线数据量很大，也不能保证闭环部署安全。

**证明思路**：

离线训练 loss 主要约束数据分布覆盖到的区域。如果部署策略进入数据未覆盖区域，模型行为变成外推，离线 loss 对该区域没有可靠约束。

**证明**：

一个典型 BC 目标可以写成：

$$\mathcal{L}_{BC}(\theta) = \mathbb{E}_{(s,a) \sim \rho_\beta}\left[\ell(\pi_\theta(s),a)\right] \tag{12.9}$$

这个目标的期望是在 $\rho_\beta$ 上计算的。

部署后真正关心的风险却更接近：

$$\mathcal{R}_{deploy}(\theta) = \mathbb{E}_{(s,a) \sim \rho_{\pi_\theta}}\left[c(s,a)\right] \tag{12.10}$$

其中 $c(s,a)$ 可以表示碰撞、失败、越界、动作不稳定等代价。

如果 $\rho_{\pi_\theta}$ 访问了 $\rho_\beta$ 很少甚至从未覆盖的区域，那么式 (12.9) 并没有充分约束这些区域的行为。

因此，降低离线 loss 不等于降低部署风险。

**这个命题告诉我们什么？**

一百万条顺利轨迹，不一定比一万条覆盖边界、失败和恢复的数据更有价值。

**常见误解**：

不要把“大数据量”等同于“大覆盖”。如果数据都来自同一种顺利场景，它们只是重复覆盖同一小片区域。

## 7. support 与 support mismatch

### 7.1 什么是 support

数据支撑集可以粗略理解为：数据实际覆盖到的区域。

记作：

$$support(\mathcal{D}) \tag{12.11}$$

如果一个状态—动作对 $(s,a)$ 在数据中出现过，或者和数据中样本非常接近，我们说它在数据支撑集内。

如果一个状态—动作对从未出现，或者离数据覆盖区域很远，它就在支撑集外。

在机械臂抓取任务中，support 可以包含：

- 工件正常摆放时的接近动作；
- 夹爪对齐后的闭合动作；
- 放入治具时的小幅调整；
- 常见光照和相机视角；
- 轻微偏差后的纠偏动作。

如果数据没有覆盖托盘变形、夹具轻微打滑、工件半遮挡、相机抖动后的恢复动作，那么这些就是支撑集缺口。

### 7.2 support mismatch

support mismatch 指训练数据覆盖区域和部署策略访问区域不一致。

可以写成：

$$support(\rho_{\pi_\theta}) \nsubseteq support(\mathcal{D}) \tag{12.12}$$

意思是：部署策略可能访问一些训练数据没有覆盖的状态—动作区域。

这时模型就会从“插值”变成“外推”。

插值是：当前输入和训练数据相似，模型在熟悉区域内做判断。

外推是：当前输入离训练数据很远，模型仍然给出一个动作，但这个动作没有数据支撑。

### 命题 12.2：support mismatch 会让离线策略在部署时自信地外推

> **命题 12.2：support mismatch 与外推风险**
>
> 当部署策略访问到 $support(\mathcal{D})$ 之外的区域时，离线训练目标无法可靠约束这些区域的动作输出。模型可能仍然输出高置信动作，但该动作缺少数据支持。

**证明思路**：

离线目标只在数据样本附近有监督信号。支撑集外没有专家动作标签或恢复动作，loss 对这些区域没有直接约束。

**证明**：

离线训练样本来自 $\mathcal{D}$。如果某个状态 $s'$ 不在数据支撑集内，那么训练集中没有足够样本告诉模型：

$$\pi_\theta(a \mid s') \quad \text{应该输出什么} \tag{12.13}$$

但神经网络在推理时不会说“我不知道”。它仍然会输出一个动作：

$$a' \sim \pi_\theta(\cdot \mid s') \tag{12.14}$$

这个动作可能看起来数值正常，甚至置信度很高，但它是从模型参数泛化出来的外推结果。

由于训练目标没有在 $s'$ 附近提供可靠约束，我们不能仅凭离线 loss 判断这个动作是否安全。

**这个命题告诉我们什么？**

离线模型最危险的地方，不一定是它明显不知道，而是它不知道自己不知道。

## 8. OOD action 如何滚雪球成 OOD state

support mismatch 不是静态问题。机器人是闭环系统，一个动作会改变未来状态。

如果模型在某个状态输出了 OOD action，下一步状态也可能被带出数据覆盖区域。

环境转移可以写成：

$$s_{t+1} \sim P(s_{t+1} \mid s_t,a_t) \tag{12.15}$$

如果 $a_t$ 不在数据支撑集附近：

$$a_t \notin support(\mathcal{D} \mid s_t) \tag{12.16}$$

那么下一步状态 $s_{t+1}$ 也可能偏离数据覆盖区域：

$$s_{t+1} \notin support(\mathcal{D}) \tag{12.17}$$

下一步模型又要在更陌生的状态上输出动作：

$$a_{t+1} \sim \pi_\theta(\cdot \mid s_{t+1}) \tag{12.18}$$

于是形成链式放大：

```text
小偏差状态
→ OOD action
→ 环境转移到更偏的状态
→ 更 OOD 的输入
→ 更不可靠的动作
→ 闭环失控
```

### 命题 12.3：OOD action 会通过环境转移放大成 OOD state

> **命题 12.3：离线模仿学习中的 OOD 滚雪球效应**
>
> 在闭环控制中，动作不仅是输出结果，也是下一步状态分布的原因。因此，某一步 OOD action 可能通过环境转移诱导后续 OOD state，使策略越来越远离数据支撑集。

**证明思路**：

只要动作影响状态转移，那么不受数据约束的动作就可能把系统带到数据未覆盖状态。新状态又会导致下一步动作继续外推。

**证明**：

假设在时间 $t$，状态 $s_t$ 仍然接近数据支撑集，但模型输出了一个数据中很少出现的动作 $a_t$。

环境按照 $P(s_{t+1} \mid s_t,a_t)$ 转移。如果 $a_t$ 使机械臂末端偏离目标、夹爪提前闭合、或车辆转向过大，那么 $s_{t+1}$ 就可能变成专家数据中很少出现的状态。

在 $s_{t+1}$ 上，离线数据没有足够样本告诉策略如何恢复。于是 $\pi_\theta$ 再次外推，输出 $a_{t+1}$。

这个过程反复发生，就会从一个小偏差变成闭环失败。

**工程含义**：

机械臂抓取时，末端稍微偏了一点并不可怕；可怕的是数据里没有“偏了以后怎么纠正”的动作。模型如果继续按正常轨迹执行，就可能越走越偏。

泊车时，车辆稍微偏离车位中心也不可怕；可怕的是数据里没有足够多“偏离后如何回正”的轨迹。模型可能继续输出在正常姿态下合理、但在偏差姿态下危险的动作。

## 9. recovery data：失败恢复数据为什么值钱

很多数据集偏爱“漂亮轨迹”。

漂亮轨迹看起来干净，训练 loss 也容易下降。但真实部署最需要的，往往不是更多完全顺利的轨迹，而是以下数据：

- 小偏差后如何纠正；
- 夹爪打滑后如何重新抓取；
- 工件姿态异常时如何调整；
- 托盘或治具轻微变形时如何重新对齐；
- 视觉短暂抖动后如何保持稳定；
- 接近危险边界时如何退出；
- 失败后如何安全停止或重新规划。

这些数据叫 recovery data，恢复数据。

### 9.1 recovery data 不是脏数据

很多团队会把失败、接管、纠偏轨迹当成“脏数据”，只保留成功轨迹。

这在纯监督学习直觉里很自然：数据越干净，loss 越好看。

但在离线模仿学习里，恢复数据是非常宝贵的。

因为它覆盖的是：

```text
策略最可能出错后会进入的区域，
以及从这些区域回到正常任务流的动作。
```

### 命题 12.4：recovery data 覆盖的是偏差后的可控区域

> **命题 12.4：恢复数据的工程价值**
>
> recovery data 的价值不在于让模型模仿失败，而在于覆盖偏差、边界和失败早期状态下的恢复动作。它可以把一部分原本属于 OOD 的状态重新纳入数据支撑集。

**证明思路**：

如果训练数据只有正常状态，那么偏差状态属于支撑集外。加入恢复轨迹后，偏差状态及其纠正动作进入数据支撑集，模型在这些状态上的行为从外推变成有监督学习。

**证明**：

假设原始数据集 $\mathcal{D}_{success}$ 只包含顺利轨迹。某个偏差状态 $s_{off}$ 不在数据支撑集内：

$$s_{off} \notin support(\mathcal{D}_{success}) \tag{12.19}$$

此时策略在 $s_{off}$ 上输出动作主要依赖外推。

如果加入恢复数据 $\mathcal{D}_{rec}$，其中包含从 $s_{off}$ 或相近状态恢复到正常任务流的动作，那么新数据集为：

$$\mathcal{D}_{new} = \mathcal{D}_{success} \cup \mathcal{D}_{rec} \tag{12.20}$$

此时 $s_{off}$ 附近的状态和对应恢复动作进入数据支撑集：

$$s_{off} \in support(\mathcal{D}_{new}) \tag{12.21}$$

于是模型在这些状态上的输出不再完全依赖外推，而可以从恢复样本中学习。

**这个命题告诉我们什么？**

恢复轨迹不是教模型失败，而是教模型失败早期如何回到正常轨道。

**常见误解**：

不要把所有失败数据直接无脑混入训练集。真正有价值的是有恢复动作、有上下文标注、有安全边界信息的失败和纠偏数据。

## 10. 离线评估能做什么，不能做什么

Offline IL 很容易让人产生一个错觉：只要验证集 loss 低，模型就能上线。

验证集 loss 有价值，但它主要回答：

> 在和数据集相似的样本上，模型输出是否接近标签？

它不能充分回答：

> 模型闭环执行后会不会进入数据集没有覆盖的状态？

因此，离线评估至少要分成几类。

| 评估类型 | 能回答什么 | 不能回答什么 |
|---|---|---|
| open-loop loss | 数据分布内动作拟合是否好 | 闭环是否稳定 |
| trajectory replay | 回放轨迹上动作是否合理 | 偏离轨迹后能否恢复 |
| coverage analysis | 数据是否覆盖关键场景 | 策略是否一定安全 |
| OOD detection | 当前输入是否远离训练数据 | OOD 后该怎么恢复 |
| closed-loop simulation | 仿真闭环表现 | sim-to-real 是否成立 |
| small-scale real test | 小范围实机风险 | 长尾场景和长期稳定性 |

对机器人学习来说，离线评估的正确姿势不是“用一个 loss 决定上线”，而是：

```text
open-loop 指标
+ 数据覆盖分析
+ 边界场景回放
+ 仿真闭环测试
+ 小范围实机灰度
+ 失败回收与再训练
```

---

## 11. 为什么第13章自然是 ACT

本章把问题推进到了一个新的位置：

```text
我们不能随便 rollout；
我们主要依赖离线数据；
离线数据存在 support mismatch 和 OOD 风险；
恢复数据很重要；
open-loop loss 不能直接代表 closed-loop success。
```

在这个前提下，下一个自然问题不是马上跳到 Decision Transformer，而是：

> 在离线数据上训练策略时，策略输出形式本身是不是也有问题？

如果策略每次只输出一步动作，它可能出现几个问题：

- 容易短视，只考虑下一步；
- 动作抖动，前后不连续；
- 难以表达“先靠近、再对齐、再夹取”的短时动作意图；
- 对遥操作数据中的延迟、噪声和小抖动敏感；
- 一步错了以后，下一步又在新的状态上继续外推。

第13章 ACT 会引入动作块：

```text
不是每次只预测一个动作，
而是一次预测未来一小段动作序列。
```

这和第12章的关系非常直接。

Offline IL 告诉我们：数据覆盖有限，闭环试错昂贵，单步 loss 不等于任务成功。

ACT 接着问：在这样的离线数据上，能不能让策略一次输出更连续、更有短时计划感的动作块，从而减少单步预测带来的抖动和短视？

这就是第12章自然引出第13章的原因。

## 12. 读完本章，你应该能判断什么

| 工程现象 | 应该想到的 Offline IL 判断 |
|---|---|
| 离线数据很多，但上线仍然失败 | 数据量不等于 support 覆盖 |
| 全是成功轨迹，验证 loss 很低 | 模型可能不会处理偏差和失败恢复 |
| 模型一旦走偏，后面越来越怪 | OOD action 通过环境转移诱导了 OOD state |
| 训练数据来自不同操作员，模型表现不稳定 | behavior policy 可能是混合策略，数据分布内部不一致 |
| 模型在正常样本上很好，在边界场景崩溃 | support mismatch 或边界数据不足 |
| 想删除失败轨迹让数据更干净 | 需要区分无效脏数据和有价值 recovery data |
| 想直接用 open-loop loss 决定上线 | open-loop loss 不能代表 closed-loop success |
| 想上 ACT / Diffusion Policy | 先检查数据覆盖，否则强模型只会更强地拟合数据偏差 |

## 13. 本章常见误解

| 常见误解 | 正确认识 |
|---|---|
| Offline IL 就是普通 BC | BC 是一种离线训练形式，但 Offline IL 更强调数据覆盖、support mismatch 和部署风险 |
| 数据越多越好 | 多只能降低采样噪声，不能自动补齐没有覆盖的风险区域 |
| behavior policy 不重要 | behavior policy 决定数据来自哪里、覆盖什么、缺什么 |
| 验证集 loss 低就可以上线 | 验证集通常仍在数据分布内，不能证明闭环安全 |
| OOD 只是输入图像不一样 | 在机器人中，OOD action 会改变未来状态，造成连锁偏移 |
| 失败轨迹都是脏数据 | 有恢复动作和上下文标注的失败轨迹非常有价值 |
| recovery data 会让模型学坏 | 合理组织的 recovery data 教模型纠偏，不是教模型失败 |
| 第12章直接通向 Decision Transformer | 更自然的下一步是第13章 ACT，因为它先解决离线数据上的动作块建模问题 |

## 16. 本章小结：从离线数据覆盖走向动作块策略

本章完成了四件事。

第一，它说明了 Offline IL 的根本约束：训练时主要依赖固定数据集，不能像 GAIL 那样频繁用当前策略 rollout。

第二，它强化了 behavior policy $\beta(a \mid s)$ 的作用。数据不是凭空产生的，而是由人类、旧模型、规则系统或混合采集流程留下的分布痕迹。

第三，它推导了 support mismatch 和 OOD action 的风险。离线 loss 只在数据覆盖区域内可靠，动作一旦出圈，会通过环境转移继续制造新的 OOD state。

第四，它强调了 recovery data 的工程价值。恢复数据不是脏数据，而是覆盖偏差后可控区域、提升闭环鲁棒性的关键数据。

但本章还没有解决一个问题：

> 在这些固定离线数据上，策略应该怎样表达动作？

如果策略每次只预测一步动作，容易短视、抖动，也很难表达连续操作意图。

因此下一章进入 ACT：一次别只想一步，机器人也需要动作小套餐。
