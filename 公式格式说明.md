# 公式格式说明

> 本文是《模仿学习的数学原理》唯一的公式格式规范入口。后续章节改造、成稿校对、公式索引整理、GitHub 网页预览与 mdBook 构建，都以本文为准。
>
> 《成书级章节改造方法.md》只保留“公式必须讲清楚”的写作要求；具体公式格式、GitHub 兼容写法和检查清单统一维护在本文中。

---

## 1. 总目标

公式规范同时服务三件事：

1. **数学表达不被改坏**：变量、上下标、希腊字母、条件概率、范数、期望、损失函数等必须保持数学含义。
2. **读者能看懂公式**：核心公式必须配套“动机—公式—读法—符号解释—直觉—工程含义—常见误解”。
3. **网页稳定显示**：公式不仅要在 mdBook 中稳定，也要尽量在 GitHub 网页直接打开 `.md` 文件时稳定显示。

一句话原则：

```text
不要只写 LaTeX 正确的公式，要写 GitHub 和 mdBook 都稳定显示、读者也能读懂的公式。
```

---

## 2. 行内公式

当前仓库正文中，行内公式优先使用美元符号：

```markdown
模仿学习要学习策略 $\pi_\theta(a \mid o)$。
```

注意：

- 不要把行内公式改成普通括号；
- 不要把公式拆成普通文本；
- 不要在公式内部随意加中文字符；
- 条件概率使用 `\mid`，不要写成普通 `|`；
- 行内公式中尽量不要写过长表达式，复杂公式放到独立公式块中。

不推荐在当前仓库中使用：

```markdown
\( \Delta x_t \)
```

因为部分页面或工具链中可能显示成普通文本：

```text
( \Delta x_t )
```

---

## 3. 独立公式

独立公式用于单独成段展示。中短公式优先使用**单行块级公式**，这样 GitHub 网页直接预览更稳定。

推荐：

```markdown
$$\mathcal{L}_{MSE}(\theta) = \frac{1}{N}\sum_{i=1}^{N}\lVert f_\theta(o_i)-a_i\rVert^2$$
```

也可以使用多行公式，但必须避免 GitHub 易误解析的写法。

---

## 4. 不要让等号单独占一行

不要写成：

```markdown
$$
\mathcal{L}_{MSE}(\theta)
=
\frac{1}{N}\sum_{i=1}^{N}\lVert f_\theta(o_i)-a_i\rVert^2
$$
```

原因：GitHub 网页预览一旦没有把该片段识别为数学块，单独一行的 `=` 可能被 Markdown 当成 Setext 标题下划线处理，导致等号消失、公式被拆散。

推荐写成：

```markdown
$$\mathcal{L}_{MSE}(\theta) = \frac{1}{N}\sum_{i=1}^{N}\lVert f_\theta(o_i)-a_i\rVert^2$$
```

或者长公式写成：

```markdown
$$
J_{BC}(\theta) =
\mathbb{E}_{o \sim d^{\pi_E}}
\left[\ell\left(\pi_\theta(o), a_E\right)\right]
$$
```

核心要求是：**等号不能单独占一行。**

---

## 5. 不要把公式编号写进 LaTeX `\tag{}`

当前已踩坑确认：GitHub 网页直接预览 `.md` 文件时，某些块级公式即使源码看起来正确，也可能因为 `\tag{...}`、短公式、prime 符号、多行公式组合而渲染异常，出现下面现象：

```text

′
(
s
,
a
)
=
α
r
(
s
,
a
)
,
α
>
0
```

典型不推荐写法：

```markdown
$$r'(s,a) = \alpha r(s,a),\quad \alpha>0 \tag{10.7}$$
```

也不推荐把大量正文公式都写成带 `\tag{...}` 的块级公式。源码可能能读，但 GitHub 网页渲染和复制结果不稳定。

推荐把公式编号放在正文标题或说明行里，公式块只放纯公式：

```markdown
**公式 (10.7)：reward 正比例缩放**

$$r'(s,a) = \alpha r(s,a),\quad \alpha>0$$
```

或者对于很短的替换式，也可以直接写在正文里：

```markdown
先考虑正比例缩放，即 **公式 (10.7)**：$r'(s,a)=\alpha r(s,a)$，其中 $\alpha>0$。
```

本书统一建议：

- 正文中用“**公式 (x.y)：公式名称**”管理编号；
- LaTeX 公式块内部不要写 `\tag{x.y}`；
- 公式索引引用正文编号，不依赖 LaTeX 自动编号；
- 短公式优先用“正文编号 + 行内公式”或“正文编号 + 单行块公式”；
- 长推导使用 `aligned`，但编号仍放在公式块上方正文中。

这样做的好处是：即使 GitHub 没有把数学块渲染好，读者也能看到公式编号、公式名称和上下文，不会因为 `\tag{}` 解析失败导致整段公式竖排或编号丢失。

## 6. GitHub 禁用宏与保守写法

GitHub 数学渲染可能禁用部分宏。当前已踩坑确认：

```text
The following macros are not allowed: operatorname
```

因此不要使用：

```latex
\operatorname
\operatorname*
```

尤其不要写：

```latex
\operatorname*{arg\,min}_{\theta}
```

推荐写成：

```latex
\arg\min_{\theta}
```

示例：

```markdown
$$\theta^* = \arg\min_{\theta}\frac{1}{N}\sum_{i=1}^{N}\ell\left(\pi_\theta(o_i), a_i\right)$$
```

---

## 7. 范数写法

向量范数优先使用：

```latex
\lVert x \rVert^2
```

不推荐：

```latex
\left\|x\right\|^2
```

原因：`\left\| ... \right\|` 在 GitHub 网页、复制粘贴或某些渲染器中容易被看成单竖线，造成绝对值与向量范数的歧义。

推荐示例：

```markdown
$$\mathcal{L}_{MSE}(\theta) = \frac{1}{N}\sum_{i=1}^{N}\lVert f_\theta(o_i)-a_i\rVert^2$$
```

---

## 8. 多行推导

多行推导用于展示一组对齐的等式。优先使用 `aligned`，但仍然要注意 GitHub 兼容性。

推荐：

```markdown
**公式 (x.y)：多行推导名称**

$$
\begin{aligned}
x &= a + b \\
y &= c + d
\end{aligned}
$$
```

注意：

- 需要换行对齐时，用 `aligned`；
- 不要用普通换行替代数学环境；
- 不要删除对齐符号 `&`；
- 如果只是一个公式，不要为了排版强行拆成多行；
- 多行公式中也不要让普通 Markdown 层面的 `=` 单独占一行；
- 公式编号放在公式块上方正文中，不要放进 `\tag{}`。

---

## 9. 不要把块级公式放进 blockquote

不推荐：

```markdown
> $$
> O(T^2\varepsilon)
> $$
```

推荐：

```markdown
> **命题：BC 的误差累积风险**
>
> 在最坏情况下，累计错误量级可能达到 $O(T^2\varepsilon)$。

$$O(T^2\varepsilon)$$
```

原因：blockquote 中的 `$$` 在 GitHub / mdBook / Markdown 插件之间兼容性不稳定。

---

## 10. 公式索引规范

每章末尾可以有公式索引，但不要把复杂公式塞进 Markdown 表格。

不推荐：

```markdown
| 编号 | 公式 | 含义 |
|---|---|---|
| (2.4) | $\frac{1}{N}\sum ...$ | MSE 损失 |
```

推荐格式：

```markdown
### 公式 (2.4)：连续动作下的 MSE 损失

$$\mathcal{L}_{MSE}(\theta) = \frac{1}{N}\sum_{i=1}^{N}\lVert f_\theta(o_i)-a_i\rVert^2$$

- **含义**：用预测动作和专家动作之间的平方距离训练确定性策略。
- **需要掌握到什么程度**：理解 MSE 适合连续动作回归，但在多模态动作下可能产生平均动作问题。
```

公式索引中的公式必须和正文公式保持一致，不允许正文和索引使用两套不同写法。

## 11. 核心公式讲解结构

成书级章节中的核心公式不能直接抛给读者，必须按下面结构讲清楚：

```markdown
**公式 (2.4)：连续动作下的行为克隆 MSE 损失**

$$\mathcal{L}_{MSE}(\theta) = \frac{1}{N}\sum_{i=1}^{N}\lVert f_\theta(o_i)-a_i\rVert^2$$

- $\theta$：……

- $o_i$：……
- $a_i$：……
- $f_\theta(o_i)$：……
- $\ell(\cdot, \cdot)$：……

直觉上，它是在做一件事：……

放到机械臂任务中，它表示：……
```

至少对每章的核心公式，要做到：

1. 公式出现前有动机；
2. 公式后解释符号；
3. 解释这个公式解决什么问题；
4. 解释工程直觉；
5. 指出常见误解。

## 12. 公式编号规范

公式编号在正文中用自然语言维护：

```markdown
**公式 (10.2)：策略的期望回报目标**

$$
\begin{aligned}
J(\pi) &= \mathbb{E}_{\tau \sim p_\pi(\tau)}
\left[
\sum_{t=0}^{T}\gamma^t r(s_t,a_t)
\right]
\end{aligned}
$$
```

不要写成：

```markdown
$$
\begin{aligned}
J(\pi) &= \mathbb{E}_{\tau \sim p_\pi(\tau)}
\left[
\sum_{t=0}^{T}\gamma^t r(s_t,a_t)
\right]
\end{aligned}
\tag{10.2}
$$
```

原因：本书优先保证 GitHub 网页预览、mdBook 构建、复制粘贴和人工审稿的一致性。正文编号比 LaTeX `\tag{}` 更稳定，也方便章节内公式索引统一管理。

- [ ] 是否有范数写成 `\left\| ... \right\|`；
- [ ] 是否有长公式被强行写成超长行内公式；
- [ ] 多行推导是否使用 `aligned`；
