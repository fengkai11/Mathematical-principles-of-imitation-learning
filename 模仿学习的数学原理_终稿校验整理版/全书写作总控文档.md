相比 v1，本版新增以下硬性要求：

1. 本书定位为**数学原理书**，公式必须讲清楚，不能只给结论。
2. 目标读者只有基础数学和基础机器学习知识，不默认熟悉概率统计、强化学习、生成模型推导。
3. 正文中每个重要公式都必须解释动机、符号、直觉、工程含义和常见误解。
4. 涉及概率统计、优化、强化学习、生成模型的基础公式，必须在附录中进行基础且详细的解释。

   - 本章公式索引；

## 1. 全书定位

### 1.1 书名

《模仿学习的数学原理：从行为克隆到 Diffusion Policy》

### 1.2 这本书是什么

这是一本面向具身智能、机器人学习和自动驾驶方向算法工程师的技术书。

它不是一本简单的论文综述，也不是一本公式堆砌型数学教材，而是一本围绕模仿学习核心问题展开的系统性技术书：

> 机器人到底如何从专家演示中学习？  
> 为什么最简单的 Behavior Cloning 会失败？  
> 为什么分布偏移是模仿学习的核心痛点？  
> 为什么现代方法从 MSE 回归动作，逐渐走向 CVAE、ACT、Diffusion Policy、Decision Transformer 和 VLA？  
> 这些方法背后的数学对象到底是什么？  
> 这些公式到底在解决什么问题？  
> 它们在真实机器人系统中又会遇到哪些工程问题？

### 1.3 这本书不是什么

本书不是：

1. **论文名词大杂烩**  
   不允许简单罗列 BC、DAgger、GAIL、ACT、Diffusion Policy、RT-2、OpenVLA 等名词，然后逐篇摘要复述。

2. **数学公式炫技集**  
   公式必须服务于理解。不能为了显得高级而塞一堆读者看完只想关电脑的符号。

3. **机器人成功学鸡汤**  
   不允许把模仿学习写成“有了数据和大模型，机器人马上进厂打工”。必须诚实讨论数据、闭环、泛化、安全、部署等问题。

4. **纯学术综述**  
   本书面向工程师和研究入门者，必须持续连接真实机器人、机械臂、自动驾驶、泊车、双臂操作、仿真实验和部署问题。

5. **默认读者概率统计很强的研究生教材**  
   不能假设读者熟练掌握 MLE、NLL、KL、ELBO、score matching、occupancy measure 等内容。首次出现必须解释，附录必须补基础。

### 1.4 全书核心主线

全书按照如下主线推进：

```text
专家演示数据
    ↓
Behavior Cloning：把模仿学习看成监督学习
    ↓
Distribution Shift：为什么监督学习视角不够
    ↓
MDP / 序列决策：机器人活在时间里
    ↓
概率策略 / 多模态动作：正确动作不止一个
    ↓
CVAE / ACT：从单步动作到动作块
    ↓
Diffusion Policy：把动作生成建模成条件去噪
    ↓
GAIL / IRL / Offline RL：从模仿动作到理解目标
    ↓
Decision Transformer / VLA：把策略学习推向大模型时代
    ↓
工程部署：让 policy 从论文视频走进真实系统
```

### 1.5 全书要回答的核心问题

每章虽然主题不同，但都应服务于以下问题：

1. 模仿学习到底在学习什么数学对象？
2. 为什么简单行为克隆容易闭环失败？
3. 分布偏移如何影响序列决策？
4. 为什么动作预测不能只用 MSE？
5. 多模态动作如何建模？
6. CVAE、ACT、Diffusion Policy 分别解决什么问题？
7. GAIL、IRL 和 Offline RL 与模仿学习是什么关系？
8. Transformer 和 VLA 给模仿学习带来了什么变化？
9. 从论文到工程部署，中间有哪些坑？
10. 一个工程师如何用这些方法设计自己的机器人学习实验？

## 3. 写作风格

### 3.1 总体风格

全书风格应满足：

```text
技术严谨 + 语言幽默 + 结构清晰 + 数学讲透 + 工程接地气
```

可以幽默，但不能油腻。  
可以类比，但不能歪曲。  
可以吐槽，但不能胡说。  
可以讲公式，但不能把读者按在黑板上摩擦。

### 3.2 叙述语气

推荐使用：

- 像一位懂机器人学习的老师在讲课；
- 像一位做过工程的研究员在复盘；
- 像一位知道读者会在哪些地方卡住的技术作者在引导；
- 偶尔像一个冷幽默段子手，但不要变成相声演员。

不推荐使用：

- 论文摘要式写法；
- 百科词条式写法；
- 过度学术腔；
- 过度营销腔；
- “某某方法横空出世，彻底解决机器人智能”这类夸张表达。

### 3.3 推荐句式

可以使用以下句式增强可读性：

1. **先拆问题**
   > 表面上看，这只是一个动作预测问题；但一旦机器人开始闭环执行，事情就开始不讲武德了。

2. **先讲直觉，再给公式**
   > 在写公式之前，我们先问一个朴素的问题：机器人到底在哪些状态上学过？又会在哪些状态上执行？

3. **公式后立刻解释**
   > 这个式子看起来有点吓人，但它其实只是在说：训练时模型看到的是专家走过的路，执行时它走的是自己踩出来的坑。

4. **工程落地提醒**
   > 论文里这个设置很干净，但真实产线不会这么配合。真实世界更像一个不写文档的甲方。

5. **批判性收束**
   > 所以，这个方法不是不好，而是它有明确适用边界。边界之外，它就会从“智能策略”变成“高维随机数生成器”。

### 3.4 每章语言密度

每章应避免连续超过 5 段纯抽象论述。  
每个核心概念至少要配：

1. 一个直觉解释；
2. 一个数学定义；
3. 一个机器人或自动驾驶例子；
4. 一个常见误区。

## 4. 幽默表达原则

### 4.1 幽默的作用

幽默不是为了搞笑，而是为了：

1. 降低读者面对数学公式的心理压力；
2. 让抽象概念更容易记住；
3. 暴露工程现实中的荒诞感；
4. 帮助读者理解方法边界。

### 4.2 推荐幽默类型

#### 4.2.1 生活类比

例如：

- Behavior Cloning 像学生刷原题；
- DAgger 像老师坐副驾；
- Diffusion Policy 像从马赛克里慢慢修出专家动作；
- VLA 像视觉、语言和动作坐到一张麻将桌上；
- Offline dataset 像驾校教材，教材没讲过鹅，路上遇到鹅就只能和鹅一起思考人生。

#### 4.2.2 工程吐槽

例如：

- “论文里的环境很安静，真实工厂里的相机支架可能正在跳广场舞。”
- “open-loop loss 很漂亮，不代表机器人不会在实机上优雅地夹空气。”
- “模型输出动作只是开始，控制器、延迟、安全边界、fallback 才是真正的职场考核。”

#### 4.2.3 反差表达

例如：

- “训练集上是天才少年，闭环执行时是第一天上路的新手司机。”
- “看起来只是预测一个动作，实际上是在改写整条未来轨迹。”

### 4.3 禁止的幽默方式

不允许：

1. 为了段子牺牲技术准确性；
2. 使用低俗、油腻、攻击性表达；
3. 把严肃数学概念完全娱乐化；
4. 在一个章节中高频重复同一类吐槽；
5. 把工程问题写成“玄学”或“玄幻小说”。

## 5. 数学推导原则

### 5.1 公式不是摆设

每个公式都必须回答：

> 它在解决什么问题？

如果一个公式不能帮助读者理解问题，就不要放。

### 5.2 重要公式五步讲解法

正文中首次出现的重要公式，必须按照“五步讲解法”说明：

```text
1. 这个公式要解决什么问题？
2. 公式中的每个符号是什么意思？
3. 这个公式直觉上在说什么？
4. 它和机器人 / 自动驾驶 / 机械臂任务有什么关系？
5. 初学者最容易误解哪里？
```

### 5.3 公式讲解框格式

正文中建议使用如下固定格式：

```markdown
### 公式拆解：为什么这个式子长这样？

公式：

\[
...
\]

它要解决的问题：

...

符号解释：

- ...
- ...

直觉理解：

...

机器人 / 自动驾驶案例：

...

常见误解：

...
```

### 5.4 公式出现前必须先讲动机

错误写法：

```text
行为克隆目标为：
L = ...
```

推荐写法：

```text
我们希望模型在看到观测 o 时，尽可能把高概率分给专家真正采取的动作 a。
为了把这个目标写成可以优化的形式，我们先把策略看成一个条件概率分布：
...
```

### 5.5 公式出现后必须解释符号

任何公式中的符号，首次出现时必须解释。例如：

\[
\mathcal L_{\mathrm{BC}}(\theta)
=
-\mathbb E_{(o,a)\sim\mathcal D}
[
\log \pi_\theta(a|o)
]
\]

必须解释：

- \(\mathcal L_{\mathrm{BC}}\)：行为克隆损失；
- \(\theta\)：模型参数；
- \((o,a)\sim\mathcal D\)：从专家数据集中抽取观测—动作样本；
- \(\pi_\theta(a|o)\)：给定观测 \(o\) 时，模型给专家动作 \(a\) 的概率；
- \(\log\)：对数；
- 负号：把最大化概率转成最小化损失；
- \(\mathbb E\)：对数据集中样本求平均。

### 5.6 正文与附录分工

正文负责：

1. 公式解决什么问题；
2. 公式如何用于当前方法；
3. 符号含义；
4. 直觉解释；
5. 工程含义；
6. 与前后方法关系。

附录负责：

1. 概率统计基础；
2. 数学推导细节；
3. 为什么 MLE 转成 NLL；
4. 为什么 MSE 来自高斯负对数似然；
5. KL、ELBO、score matching 等基础解释。

### 5.7 禁止偷懒表达

以下表达禁止出现：

- “显然”
- “容易得到”
- “不难证明”
- “省略推导”
- “读者可自行验证”
- “这个公式很简单”
- “大家都知道”

替代表达：

- “我们一步一步拆开来看。”
- “这一步背后的直觉是……”
- “如果对概率基础不熟，可以先看附录 C。”
- “这里先给结论，完整推导放在附录 D。”

## 6. 工程案例原则

### 6.1 案例来源

优先使用：

1. 机械臂抓取；
2. 双臂操作；
3. 自动驾驶；
4. 自动泊车；
5. 移动机器人导航；
6. 工厂装配；
7. 具身智能中的语言条件操作。

### 6.2 案例必须服务概念

工程案例不是装饰品，必须用于解释：

1. 为什么这个数学对象出现；
2. 这个 loss 在真实任务中怎么用；
3. 这个方法的边界在哪里；
4. 为什么论文结果迁移到工程中会遇到问题。

### 6.3 每章至少包含 2 个工程案例

建议每章至少包含：

1. 一个机器人 / 机械臂案例；
2. 一个自动驾驶 / 泊车 / 移动机器人案例。

### 6.4 必须有工程风险分析

每章都要至少有一小节回答：

> 这个方法在论文里看起来有效，为什么工程中可能失败？

## 13. 章节审稿标准

每章审稿必须检查以下项目：

### 13.1 数学审稿

1. 符号是否统一；
2. 公式是否正确；
3. 公式是否跳步；
4. 概率统计概念是否解释；
5. 是否给出附录引用建议。

### 13.2 工程审稿

1. 案例是否贴近机器人 / 自动驾驶 / 机械臂；
2. 方法边界是否讲清；
3. 工程风险是否真实；
4. 是否避免“论文即产品”的误导。

### 13.3 风格审稿

1. 是否幽默易读；
2. 是否有过度段子化；
3. 是否有过度学术腔；
4. 是否有废话；
5. 是否适合目标读者。

### 13.4 配图审稿

1. 图是否讲清一个概念；
2. 图中公式是否清晰；
3. 图中中文是否无乱码；
4. 图是否过度拥挤；
5. 图是否和正文配合。

7. 图编号是否连续；

## 15. 禁止出现的问题

1. 公式裸奔：只给公式，不解释动机和符号；
2. 直接堆论文名词；
3. 过度吹捧某个方法；
4. 把 BC、ACT、Diffusion Policy 写成互相替代的线性升级关系；
5. 忽视数据问题；
6. 忽视闭环执行；
7. 忽视工程部署；
8. 把 VLA 写成万能机器人智能；
9. 用“显然”“容易得到”等偷懒表达；
10. 幽默过度影响严谨；
11. 每章配图只是装饰而不是解释；
12. 附录缺失概率统计基础解释；
13. 不区分 state 和 observation；
14. 不区分 open-loop 与 closed-loop；
15. 不区分训练 loss 和任务成功率。
